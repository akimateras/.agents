# 日本語技術文書の表記規約

## 基本方針

- 読者が操作、判断、検証するために必要な情報を優先する。
- 正確性、完全性、検索性に加え、語句の関係を一読で理解できる日本語を必要条件とする。
  不自然さによって修飾先や動作主体が曖昧になる文章を、正確な文章とみなさない。
- 説明できない事実や理由を補完しない。
- 不要な前置き、一般論、宣伝表現を入れない。
- 依頼に関係しない句読点、空白、見出し、表記を機械的に変更しない。

## 事実の根拠と表記の根拠を分ける

表記と文体は、次の順で根拠を優先する。

1. リポジトリの正式な用語集と表記規約
2. 対象文書および関連文書にある、現在も有効で人が確認した自然な日本語表記
3. この規約の既定方針

実装、差分、スキーマ、テストは、挙動や名称の事実を確認する根拠として使う。
英語のソースコードや差分に現れる語句は、一般概念を日本語でどう書くかの根拠にはしない。
コード上で英単語が使われていることだけを理由に、その英単語を日本語の文へ移さない。

既存の日本語は、正式な表記規約、用語集、または人が確認した文書だと判断できる場合に表記の根拠として使う。
生成経緯や品質を確認できない文章は、繰り返し使われていても正しい表記とはみなさない。
周辺文書の不自然な表現を一貫性のために増やさない。

## 意味から日本語を組み立てる

原文、差分、コードから直接文章を作らず、内部作業として次の順序で組み立てる。

1. 記載する事実を、主体、対象、動作、条件、結果、因果関係に分ける。
2. 各語句を、原表記が必要な名称と、日本語で説明する一般概念に分ける。
3. 原文の語順と単語のまとまりには従わず、事実同士の関係だけから日本語の文を作る。
4. 原表記が必要な名称だけを文へ戻す。
5. 完成した文が元の事実を過不足なく表すか照合する。

英語で先に要約してから日本語へ翻訳しない。
英語版も必要な場合は、同じ事実の整理から英語と日本語を別々に作る。

## 原表記が必要かを分類する

文中の語句が次の対象そのものを指す場合だけ、正式な原表記を維持する。

- ソースコード上の識別子
- コマンド、オプション、環境変数
- API、プロトコル、規格、製品の正式名称
- UI に表示される正式なラベル
- ログおよびエラーメッセージ
- リポジトリの用語集で表記が指定された語
- ユーザーまたは正式な仕様で定義された新しい名称

識別子を指す場合は、原則としてインラインコードで示す。
同じ綴りがコードに存在しても、文中で一般概念を説明している場合は識別子として扱わない。
正式名称と一般的な説明を区別し、正式名称には必要に応じて平易な日本語説明を加える。

原表記を維持する対象に該当しない語句は、次のように扱う。

- IT 分野で定着した外来語は、一般的なカタカナ表記にする。
- 日本語で定着した概念、動作、状態、条件は、日本語で書く。
- 英語では名詞で表される操作でも、日本語で条件や時点を表す場合は動詞を使う。
- 定着した略語は維持してよいが、略語に付く一般語まで英字のまま残さない。

例えば、正確な識別子を指す `messageRenderKey` は原表記を維持するが、概念を説明するなら「メッセージの描画キー」と書く。
`first normal open` のような英語の名詞句は語順を写さず、意味が「通常表示を最初に開いたとき」なら、その条件を日本語の節で表す。

## 独自表現を作らない

一般的な動作、状態、判断、説明に使う普通名詞や動詞は、既存の日本語表現を優先する。

根拠がなければ次の表現を追加しない。

- 英単語を連結した独自表現
- 一般的でない英単語の無説明使用
- 意味を確認していないカタカナ転写
- 同一概念に対する新しい別名
- 英語らしく見せるためだけの名詞化
- リポジトリ内に存在しない略語

定着した日本語表現を確認できない場合は、原語を残し、直後に平易な日本語で意味を説明する。

プログラミングや IT 分野で広く定着した用語は、一般的な表記を使う。
日本語へ置き換えるためだけに不自然な訳語を作らない。
対象読者に伝わりにくい場合は、初出で意味を補う。

## 語句の関係を明示する

- 複数の英単語からなる句を、まとまりのまま日本語の助詞へ接続しない。
- 修飾語がどの語にかかるかを決め、必要なら助詞、動詞、節を補う。
- 中黒やスラッシュを、説明していない関係の代わりに使わない。
- 「A の B」が条件、所有、対象、所属のどれを表すか曖昧なら、具体的な動詞で書き換える。
- 一般概念を表す英字と日本語を連結した複合語を作らない。
- 抽象名詞と「行う」「対応する」「成立させる」だけで変化を表さず、観測できる動作や結果を書く。定義済みの設計概念を指す場合だけ、その抽象名詞を維持する。

情報を圧縮するときは、重要度の低い事実を省く。
助詞や動詞を削って英語の名詞句へ置き換えたり、複数の事実を一文へ詰め込んだりしない。
1 つの文を複数の文に分けてよい。

## 情報を短くする順序

文章を短くする必要がある場合は、文書の目的を損なわない範囲で、次の順に削る。

1. 挨拶と目的に不要な導入
2. 一般論
3. 重複
4. 背景と歴史
5. 補助的な例
6. P1 情報（誤用防止、保守、正確な理解に必要な情報）

P0 情報（利用、設定、検証、安全な運用、障害対応に不可欠な情報）は削らない。
P0、P1、P2 の定義の正本は `SKILL.md` の「必要な情報を整理する」にあり、丸括弧内の説明はその要約である。

## 表記の既定

リポジトリの表記規約と、根拠にできる既存文書のどちらからも決められない場合に適用する。

- 文体は対象文書および関連文書に合わせる。合わせる先がない場合はである調で書く。
- カタカナの長音は省略しない。「サーバー」「ユーザー」と書き、「サーバ」「ユーザ」としない。
- 英数字は半角で書く。「ＡＰＩ」「１２３」のような全角英数字を使わない。
- 半角英数字およびインラインコードと日本語の間に半角スペースを入れる。句読点や括弧に隣接する位置には入れない。
- 句読点は「、」「。」を使う。
- 日本語文中の丸括弧は全角の「（」「）」を使う。
- 数量は「3 つの方法」「1 回」のように半角算用数字で書く。「一時的」「一部」など、数を意味しない慣用語は漢数字のまま使う。
- 数値と単位記号の間には半角スペースを入れず、「50%」「10GB」「500ms」のように書く。単位記号まで含めた数値表現の全体を半角英数字とみなし、日本語との間の空白規則を適用する（例: 50% を超える）。
- 補助的な用法の語はひらがなで書く。「〜してください」「〜できる」「〜すること」「〜するとき」「〜のため」と書き、「下さい」「出来る」「事」「時」「為」を使わない。この列挙は例であり、「さらに」「および」「ただし」など、実質的な意味の薄い接続詞や副詞も同様にひらがなで書く。
- 箇条書きの項目は、文なら「。」で終え、名詞句なら句点を付けない。同じリスト内では形を揃える。
- 日付は文中では「2026 年 7 月 23 日」のように書く。表、ログ、ファイル名では ISO 8601 形式（2026-07-23）を使ってよい。
- ラベルや見出しの区切りに使うコロンは半角の「:」で書き、直後に半角スペースを入れる。
- 感嘆符と疑問符は原則使わない。使う場合は全角の「！」「？」で書く。
- 数値や日付の範囲は全角の「〜」で書き、日本語文中で半角の「~」を範囲の意味に使わない。コマンド、パス、コードに現れる「~」は原表記を維持する。
- 日本語文中の引用と強調には「」を使い、その内側で引用を重ねる場合は『』を使う。半角の引用符（"" と ''）は、コードや英語の原表記にだけ使う。
- 段落内の改行の粒度は、文書全体で統一する。

## 文体

- 結論または操作目的を先に書く。
- 1 つの段落では 1 つの論点を扱う。
- 誰が何をするかを省略しすぎない。
- 原文やソースコードの語順を写さず、意味と論理関係を理解して自然な日本語に組み立て直す。
- 抽象語だけで説明せず、読者が確認できる具体的な対象、動作、条件を書く。
- 「適切に」「柔軟に」「シームレスに」など、判定条件のない形容を避ける。
- 「〜することが可能です」より「〜できます」を優先する。
- 同じ語を不自然に言い換えず、用語を固定する。
- 条件、操作、結果を同じ文に詰め込みすぎない。

## 書き換えの例

この規約が防ぐ典型的な失敗と、その書き換え先を示す。

- 英語の名詞句を助詞へ直結しない。
  - 悪い例: connection poolのexhaustion時にリトライする
  - 良い例: 接続プールが枯渇したときに再試行する
- 定着を確認していないカタカナ転写を使わない。
  - 悪い例: 設定をパーシストする
  - 良い例: 設定を永続化する
- 一般概念を表す英字と日本語を連結した複合語を作らない。
  - 悪い例: retry回数を設定する
  - 良い例: 再試行回数を設定する
- 原文の語順を写さず、修飾関係を組み立て直す。
  - 悪い例: ロックアウト前の残っている再試行の数を返す
  - 良い例: ロックアウトまでの残り再試行回数を返す
- 抽象名詞と「行う」で動作をぼかさない。
  - 悪い例: 入力値のバリデーション処理を行う
  - 良い例: 入力値を検証する
- 動作主体を省略して受け身を連ねない。
  - 悪い例: 設定が読み込まれた後、キャッシュが更新され、通知される
  - 良い例: サービスは設定を読み込んだ後にキャッシュを更新し、利用者へ通知を送る

## 完成後の日本語監査

本文案を作った後、次の順序で全文を読み直す。

1. 日本語文中の英字表記を 1 件ずつ確認する。原表記が必要な対象なら、正式な綴りか、識別子にインラインコードが付いているかを確認する。それ以外は定着したカタカナ表記または日本語へ直す。
2. 英語の複数語が日本語の助詞へ直接つながっていないか確認する。
3. 連体修飾、「の」、中黒、スラッシュの各関係を言葉で説明できるか確認する。
4. 各文の主体、対象、動作を特定し、条件と結果の修飾先を確認する。
5. 原文や根拠を参照できない読者を想定して読み直し、技術的な前提を知る読者が一度で意味を取れるか確認する。
6. 原文または根拠と照合し、自然さを優先した結果として意味を追加、削除、変更していないか確認する。
