# コメントチェックリスト

## 先に確認すること

- 対象言語のドキュメンテーションコメント形式を確認する。
- Javadoc、KDoc、Rustdoc、Go Doc、TSDoc など、プロジェクトで採用している生成・検査ツールの規約に従う。
- コメントの言語と用語は、ユーザーの依頼およびプロジェクトの方針に従う。
- 既存コメントのタグ、句読点、文体を正当な理由なく混在させない。

## コメントを書く対象

次のいずれかをコードだけでは明確に表現できない場合に書く。

- なぜこの実装が必要か
- 守るべき不変条件
- 非自明な前提条件
- 単位、時刻、タイムゾーン
- 所有権、ライフタイム、解放責任
- 並行処理、排他、スレッド安全性
- 外部仕様との互換性
- 副作用
- 例外条件
- 性能上の制約
- 一見不要に見える処理を残す理由
- 回避策と、その回避策を削除できる条件

## 書かないコメント

- コードを日本語に読み替えただけの説明
- メソッド名や変数名の繰り返し
- 実装から確認できない設計理由
- 将来の予定を断定するコメント
- 「ここで処理する」など情報量のない説明
- コード変更によってすでに成立しなくなった説明

## 公開 API コメント

内部アルゴリズムではなく、利用者が守る契約を説明する。
必要に応じて次を含める。

- API の目的
- 引数の意味、有効範囲、null 許容性
- 戻り値と特殊値
- 例外またはエラー
- 副作用
- 非同期処理とキャンセル動作
- スレッド安全性
- 所有権
- 非推奨となった API と移行先
- 非自明な使用例

## 検査

- コメントが現在のシグネチャと実装に一致する。
- 引数名、戻り値、例外、既定動作がコードと一致する。
- 生成ツールのタグやリンクが有効である。
- 利用可能なドキュメンテーションコメント用の lint または文書生成を実行する。
- コメントを追加したことで、コードより保守しにくい重複を作っていない。
