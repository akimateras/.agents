# README チェックリスト

## 役割と対象

README は、初見の読者が対象を理解し、次の行動を選ぶための入口として扱う。
実装者の作業記録、検証結果の保管場所、内部設計の網羅的な説明にはしない。

リポジトリの種類と README の主な読者を先に判定する。
複数の種類に該当してもよい。

- ライブラリ
- CLI
- サービス
- GUI アプリケーション
- 開発ツール
- サンプルまたはテンプレート
- 複数の成果物を含むリポジトリ

利用者向け情報と開発参加者向け情報を区別する。
主な読者が README を読んで最初に判断することを特定する。
該当しない節を機械的に追加しない。
既存 README を大きく改訂する場合は、残すべき利用者向け情報と重要な制約を先に抽出する。

## 冒頭と構成

冒頭だけを読んだ人が、少なくとも次の内容を説明できるようにする。

- 対象が何であり、何のために存在するか。必要ならシステム内の位置付け
- 対象が何を提供し、利用者が何をできるか

概要と提供機能の後は、読者の行動順を基準に必要な情報を配置する。

1. 現在利用できる範囲と、未実装、試験中、廃止予定などの状態
2. 前提条件、導入方法、最小の実行可能な使用例
3. 設定、主要な入出力、他の構成要素との接続方法
4. 運用上の注意、既知の制約、問題調査の入口
5. 設計資料、API 定義、変更履歴などの詳細情報への参照

対象と読者に合わない項目は省略し、見出しを機械的に固定しない。

## P0 候補

対象の利用方法に必要な項目だけを選ぶ。P0 は、利用、設定、検証、安全な運用、障害対応に不可欠な情報を指す（定義の正本は `SKILL.md` の「必要な情報を整理する」）。

- このソフトウェアが何をするか
- 対象読者または利用者
- 現在利用できる範囲と、未実装、試験中、廃止予定などの状態
- 対応環境、対応バージョン、前提ソフトウェア
- インストール方法
- 最小の実行可能な使用例
- 必須設定
- 設定名、値の形式、必須性、既定値
- シークレットを安全に設定する方法
- 実行結果または成功確認方法
- 重大な制約
- ライセンスと利用条件（公開リポジトリなど、利用判断に必要な場合）
- 発生頻度の高いエラーと復旧方法
- 既存利用者に影響する更新または移行手順

実際のシークレット、個人情報、本番環境固有の値を例へ含めない。

## 開発者向けの場合の候補

開発参加者が主な読者の場合は、次の項目も P0 候補として扱う。

- リポジトリ構成
- 開発環境の準備
- ビルド
- テスト
- lint、format、型検査
- 生成物の更新方法
- コントリビューション手順

## 検査

- README 中のコマンドが実在し、必要な作業ディレクトリと前提条件が分かり、記載された文脈で使用できる。
- オプション名と環境変数名が実装と一致する。
- 既定値がコードまたは設定定義と一致する。
- 現在の機能、将来の予定、廃止された挙動を区別できる。
- プレースホルダーと実値を区別できる。
- サンプルが自己完結しているか、必要な前提条件が明記されている。
- ローカルで安全に実行できるサンプルは、コピーして実行できる。
- 実行できないサンプルは、静的に照合し、未確認事項を報告する。
- リンク、アンカー、ファイルパス、参照先の節が存在する。
- 文書の変更だけで、公開インターフェース、対応範囲、将来計画を新たに確約したように見えない。
- 実装日誌、検証結果、利用に不要な内部設計や一般論が、利用手順や主要な提供機能より長くなっていない。

コマンドの実行可否は、`SKILL.md` の「安全かつ必要十分に検証する」に従って判断する。

## 読者視点の検査

README だけを渡された初見の読者として、次を確認する。

- 対象の目的と主要な提供機能を、冒頭から短時間で把握できる。
- 現在使えるものと、まだ使えないものを区別できる。
- 試すための最短経路と、詳しく調べるための参照先が分かる。
- 重要でない詳細が、利用判断や正しい利用に必要な情報を埋もれさせていない。

作成または大幅改稿で満たさない項目がある場合は、見出しを追加するだけでなく、情報の順序と分量を見直す。
