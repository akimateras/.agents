# 文書種別チェックリスト

対象文書の目的に該当する節だけを使う。
1 つの文書が複数の種別を兼ねる場合は、必要事項を統合し、重複させない。

## 開発者ガイド、チュートリアル、操作手順

- 対象読者と達成する結果
- 適用条件と前提条件
- 必要な権限、依存関係、入力データ
- 順序が明確な操作
- 各段階の期待結果
- 完了を確認する方法
- 現実的な失敗経路と復旧方法

## リファレンス

- 対象範囲と対応バージョン
- 用語とデータ形式
- 公開インターフェースと利用契約
- 必須性、既定値、許容値、境界値
- エラーと副作用
- 関連する操作手順へのリンク

リファレンスでは網羅性を優先し、操作手順や背景説明を混在させすぎない。

## 設計文書と ADR（Architecture Decision Record）

- 状態、対象範囲、決定日
- 解決する問題と制約
- 採用した決定
- 検討した主要な代替案
- 採用理由と不採用理由の根拠
- 利点、欠点、運用上の影響
- 未解決事項

現行仕様、提案、過去の決定を区別する。
リポジトリで確認できない設計理由を補完しない。

## 運用手順と障害対応手順

- 実行を開始する条件と観測された症状
- 影響範囲と優先度
- 必要な権限、安全条件、バックアップ
- 診断手順と判断分岐
- 操作ごとの期待結果
- 復旧確認、切り戻し、エスカレーション条件
- 記録すべきログまたは証跡

本番環境の操作を、文書検証のために実行しない。

## 移行手順

- 移行元と移行先のバージョンまたは状態
- 対象者と適用条件
- 破壊的変更と互換性への影響
- 事前検査とバックアップ
- 順序が明確な移行操作
- 停止時間または性能への影響
- 移行後の検証
- 失敗時の切り戻し条件と手順

## リリースノートと変更履歴

- 対象バージョンとリリース日
- 破壊的変更と、影響を受ける利用者
- 新機能、変更、修正、非推奨化の区分
- 更新前に必要な操作と、移行手順への参照
- 既知の問題

利用者への影響が大きい変更から順に書く。
利用者に影響しない内部実装の変更を、利用者向けの変更と同じ扱いで並べない。
