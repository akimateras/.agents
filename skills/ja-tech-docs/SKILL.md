# 日本語技術文書

## 目的

依頼範囲を守り、確認できる根拠とプロジェクト固有の用語に基づいて、検証可能で実用的な日本語技術文書を作成、改訂、レビューする。

## 参照ファイル

- 常に [style-ja.md](references/style-ja.md) を読む。
- README を扱う場合は [readme-checklist.md](references/readme-checklist.md) を読む。
- ドキュメンテーションコメントまたはコードコメントを扱う場合は [comment-checklist.md](references/comment-checklist.md) を読む。
- 開発者ガイド、リファレンス（API ドキュメントを含む）、設計文書、運用手順、移行手順、リリースノート、変更履歴を扱う場合は [document-type-checklist.md](references/document-type-checklist.md) を読む。
- 日本語化または他言語版との同期を扱う場合は [sync-checklist.md](references/sync-checklist.md) を読む。

各チェックリストの構成や項目とリポジトリの慣例が食い違う場合は、リポジトリの慣例を優先する。

## 動作モード

依頼内容に応じて次のいずれかで動作する。

- 作成・改訂:
  対象ファイルを編集する。検証はワークフローの「安全かつ必要十分に検証する」に従って行う。
- 本文案の提示:
  ファイルを変更せず、回答内に案を示す。
- レビュー・監査・診断:
  ファイルを変更せず、根拠と場所を示して指摘する。

編集の依頼だと確定できない場合は、ファイルを変更しないモードを選ぶ。
依頼が問題の指摘や評価を求めているならレビュー・監査・診断を、文章そのものを求めているなら本文案の提示を選ぶ。
日本語化（翻訳）と他言語版との同期は、作成・改訂として扱う。

すべてのモードに次の制約を適用する。

- 対象ファイルを特定できない場合は、リポジトリ内を調査する。調査しても特定できない場合は、推測で編集しない。
- 依頼範囲を超える構造変更、仕様変更、全面的な書き換えを行わない。

## ワークフロー

### 1. 対象、範囲、根拠を確認する

対象文書、対象読者、利用目的、文書種別、変更範囲を特定する。
リポジトリの指示、用語集、表記規約、正本となる文書があれば確認する。
新規文書は、配置場所とファイル名をリポジトリの慣例から決める。慣例が見つからない場合はユーザーに確認する。確認できない場合は、候補の配置場所と選定理由を示して仮置きし、最終報告で明示する。

各事実は、その事実を定義する情報源で確認する。

- 公開インターフェースは、公開 API またはスキーマで確認する。
- 設定名、値の形式、必須性、既定値は、設定定義で確認する。
- 挙動、例外、副作用は、実装およびテストで確認する。
- 対応環境と対応バージョンは、ビルド設定や正式なサポート方針で確認する。
- テストと実行可能なサンプルは、記載内容の裏付けとして使う。
- 既存文書だけを、実装上の事実の根拠にしない。

日本語の表記と文体の根拠は、[style-ja.md](references/style-ja.md) の「事実の根拠と表記の根拠を分ける」に従って判断する。

情報源が矛盾する場合は、その事実を定義する情報源を追加調査する。
解消できない矛盾を黙って選択または混在させない。

コマンド、オプション、既定値、対応バージョン、例外、副作用、設計理由を推測しない。
重要な事実を確認できない場合は、追加調査するか、断定を避ける。
確認できないまま成果物に残る事項は、未確認として報告する。

### 2. 必要な情報を整理する

複数の節にまたがる改訂、説明不足の監査、または新規文書の作成では、内部作業用として次の一覧を作る。単一箇所の小さな修正では省略してよい。

| 優先度 | 必要な情報 | 根拠 | 記載場所 | 状態 |
|---|---|---|---|---|

「根拠」には事実を確認した情報源を、「記載場所」には文書内で記載する節または位置を書く。
「状態」には「確認済み」「未確認」「範囲外」のいずれかを書く。

優先度は次の基準を使う。

- P0:
  利用、設定、検証、安全な運用、障害対応に不可欠な情報。
- P1:
  誤用防止、保守、正確な理解に必要な情報。
- P2:
  背景、歴史、比較、補足的な説明。

依頼範囲内の情報は P0、P1、P2 の順に扱い、P0 が未記載のまま P2 を詳述しない。
範囲外で見つけた P0 は変更せず、最終報告で簡潔に言及する。

### 3. 作成、改訂、レビューを行う

- 妥当な既存構造、著者の意図、周辺の文体を維持し、差分を必要最小限にする。
- 本文の組み立て、文体、表記は [style-ja.md](references/style-ja.md) に従う。
- 新しい節は、読者が情報を探しやすくなる場合だけ追加する。
- 手順には、適用条件、前提条件、正確な操作、期待結果、検証方法を含める。
- 現実的な失敗経路がある場合は、症状、原因、復旧または切り戻し方法を含める。
- 同じ内容を概要、本文、注記で繰り返さない。
- レビューでは、重要度、ファイルと位置、根拠、影響、修正案を示す。重要度の定義は「最終報告」にある。

### 4. 安全かつ必要十分に検証する

対象文書と依頼範囲に関係する検証だけを行う。
静的な照合、既存の検査、読み取り専用操作、ドライラン（変更を適用しない試行）を優先する。

コマンドを実行する前に、次を確認する。

- ファイルやデータを変更または削除しないか
- 外部サービスや本番環境の状態を変更しないか
- 追加の権限、認証情報、ネットワーク接続、費用を必要としないか

ユーザーが依頼した範囲に含まれない外部状態の変更は、文書検証だけを理由に実行しない。
安全に実行できない操作は、実装、設定、`--help`、ドライランなどで照合し、確認できなければ未確認として報告する。

利用可能な範囲で次を行う。

- Markdown の構文検査
- リポジトリにある文書用 lint 設定（textlint、markdownlint、prh 辞書など）の実行
- リンク、アンカー、ファイルパスの確認
- コマンド、オプション、設定名、既定値、環境変数名の照合
- 安全に実行できるサンプルまたはスモークテスト
- 用語集と表記規約との照合
- [style-ja.md](references/style-ja.md) の「完成後の日本語監査」の実施
- 変更差分のレビュー

### 5. 完了を判定する

依頼範囲と文書種別に適用される項目について、次を確認する。

- 依頼範囲内の P0 が記載されている。
- 根拠のない仕様説明や、エージェントが独自に作った名称がない。
- 現行の仕様、将来の予定、過去の挙動が区別されている。
- コマンド、設定、リンクは確認済みか、未確認であることが報告されている。
- 同じ概念の表記が統一されている。
- [style-ja.md](references/style-ja.md) の「完成後の日本語監査」を完了している。
- コメントがコードの逐語説明になっていない。
- 重要でない背景が、利用手順や利用契約より詳しくなっていない。
- 実施可能で関係する検証を完了している。

未確認事項が成果物の正否または安全性を左右する場合だけ、完了を保留する。
それ以外の未確認事項は明示して完了できる。

## 最終報告

編集を行った場合は、次を簡潔に報告する。

- 変更したファイル
- 補った重要情報
- 実行した検証と結果
- 確認できなかった事項

本文案を提示した場合は、提示した案について同じ項目を報告する。「変更したファイル」の代わりに、案の反映先として想定するファイルを示す。

レビューだけの場合は、次の重要度の高い順に指摘を示す。

- 高: 読者が操作、設定、判断を誤る原因になる誤りまたは欠落
- 中: 誤解、誤用、保守性の低下を招く問題
- 低: 表記や文体の揺れ

各指摘には次を含める。

- ファイルと位置
- 欠落または誤り
- 根拠
- 影響
- 修正案

同種の低重要度の指摘が多数ある場合は、1 件ずつ列挙せず、揺れのパターン、代表例、件数でまとめてよい。

指摘がない場合は、確認した範囲と検証内容を示したうえで、重大な指摘がないことを明記する。
