# Longneck親子作業契約

## 目次

- 親の責務
- ユーザー判断とブロッカー解除
- 子へ渡すプロンプト
- 子の責務
- 子の終了報告
- 親による照合
- 無進捗と終了

## 親の責務

親だけがtitle、次の作業単位、ループ継続を決める。一度に起動する子は1つにし、
毎作業単位で同じモデルを継承する新しい子を使う。子を再利用しない。子が起動できる
サブエージェントは、ファイルを変更しない読取専用の探索に限る。
対象titleの助言的ロックはループ全体を通して親が保持し、子は取得しない。

親は起動直前のGit状態とLongneck fingerprintを記録し、終了後に実ファイルを照合する。
子の自己申告だけで完了、無進捗、変更範囲を判断しない。

## ユーザー判断とブロッカー解除

`判断待ち` と `ブロック中` の解除は親だけが行い、子へ委譲しない。ループの開始前と、
`state.md` を読み直すたびに次を適用する。

- ユーザーが `判断待ち` の問いへ明示的に回答した場合は、回答を該当タスクの目的・
  完了条件へ反映し、区分を `実行待ち` または `調査待ち` へ移す。回答がどの問いへの
  ものか一意に特定できなければ、推測せず聞き返す。
- `ブロック中` は、記録された再開条件を読み取り専用の確認で検証できる場合に限り
  親が検証し、解消の証拠が得られたタスクだけを `実行待ち` または `調査待ち` へ移す。
  証拠なしに移さず、ブロッカーの解消作業自体を親は行わない。
- 移動のたびに `state.md` の全体状態と更新日時を同期し、`progress.md` の `作業ログ`
  先頭へ `管理作業` の記録を1件追加して、根拠（ユーザー回答または確認した証拠）を残す。
- この編集は `state.md` と `progress.md` の管理作業記録に限る。目的・スコープ・
  完了条件の変更を伴う回答は、先に [amend.md](amend.md) の契約で
  `description.md` を改訂する。
- 自分の管理編集の後はCLIの `validate` を通し、基準として記録するfingerprintを
  取り直す。以後の同時変更検出は新しい基準と比較し、自分の編集を他プロセスの
  変更と誤認しない。

## 子へ渡すプロンプト

少なくとも次を渡す。対象タスクの全文には、CLIの `tasks` が報告する該当task IDの
`body` をそのまま使い、`state.md` から手で転記しない。

```text
$longneck-head を使い、次の一作業単位だけを担当してください。

Longneck root: <absolute-root>
title: <exact-title>
task ID: <T-0001>

対象タスク:
<CLIの tasks が報告する該当タスクの body（state.mdにあるタスク項目の全文）>

作業対象と正本:
- <path and role>

コミット方針:
<ユーザー指示とプロジェクト固有ルール>

制約:
- 適用されるAGENTS.mdとユーザー指示を読む。
- サブエージェントは読取専用の探索に限り、実装や文書変更を委譲しない。
- 次タスクを選ばず、ループを継続しない。
- CLIの lock を取得しない。対象titleのロックは親が保持している。
- 既存変更を保持する。
- ユーザー判断を推測しない。
- 実装、関係文書、検証、Longneck記録を同じ作業単位で同期する。
- .longneck外に、Longneckのパス、title、内部IDへの逆参照を作らない。
- 終了報告は worker-contract.md の「子の終了報告」の形式で返す。
```

## 子の責務

子は次を行う。

1. `longneck-head` の必須契約を読み、root/title/task IDを再検証する。
2. 対象タスクが現在も `調査待ち` または `実行待ち` にあり、依存関係を満たすことを
   確認する。ずれていれば変更せず報告する。
3. 一作業単位の実装、調査、文書更新、関係する検証を行う。
4. 現在必要な残作業を `state.md`、設計判断と履歴を `progress.md`、現在の目的には
   不要な将来知見を `issues.md`、詳細記録を `docs/` へ分類する。恒久的な製品知識は
   Longneckへ複製せず、既存の外部正本へ置く。
5. 完了タスクを `state.md` から除き、残タスクの `依存関係` から完了により満たされた
   参照を取り除き、snapshot、全体状態、`更新日時` を現在へ同期する。
6. 外部成果物、コメント、文書、コミットメッセージに、Longneckのパス、title、
   内部IDへの逆参照を作らない。
7. プロジェクト方針に従ってコミットし、結果を報告する。方針がなければ独断で
   `.longneck` の追跡状態を変更しない。

一作業単位で完了へ到達できない場合は、終了前に、残作業の分解、ブロッカー、または
必要なユーザー判断を `state.md` へ記録する。「無進捗と終了」に従い、この記録がない
未完了のtask IDは再委譲できず、親は全体ループを停止する。

`description.md` は変更しない。目的・スコープ・完了条件の改訂は、親が
[amend.md](amend.md) の契約で行う。`longneck-persist`、`longneck-flush`、
`longneck-sever`、`longneck-amend`、`longneck-repair`、全体オーケストレーションを
行わない。

## 子の終了報告

次のキーをこの順で返す。

```text
status: completed | partial | needs_user | blocked | no_progress | failed
task_id: <T-0001>
summary: <短い要約>
changed_files:
  - <path>
longneck_updates:
  - <state/progress/issues/docsの更新>
verification:
  - <確認と結果>
commits:
  - <sha and title、またはnone>
unrecorded_findings:
  - <記録不能な知見と理由、またはnone>
user_decisions:
  - <問い、選択肢、pros/cons、またはnone>
remaining_risks:
  - <残るリスク、またはnone>
```

完了条件または検証を満たさなければ `completed` を使わない。変更を残せなかった場合は
理由を `unrecorded_findings` に書く。通常の成功時は永続的知見を文書へ分類し、
`unrecorded_findings` を `none` にする。

## 親による照合

子の終了後、親は次を確認する。

1. 起動前後の `HEAD`、Git状態、Longneck fingerprintを比較する。
2. 子が挙げたファイルと実差分を照合し、既存変更や対象外変更を分離する。
3. 対象タスクの完了条件と検証結果を確認する。必要なら安全な検証を再実行する。
4. `description.md`、`state.md`、`progress.md`、`issues.md` と関係する `docs/` を
   読み直し、現在状態、履歴、知見の分類を確認する。
5. 外部変更と新規コミットに、Longneckのパス、title、内部IDへの逆参照が
   ないことを確認する。
6. 根拠が十分な記録漏れだけを親が補う。実装の修正や同じタスクの再実行で
   子の失敗を隠さない。

子がコミットした場合も起動前 `HEAD` から確認し、成果を見落とさない。
子が作成後に変更を取り消した場合、対象外の一時ファイルだけを残した場合、報告だけで
ファイルを変更しなかった場合は無進捗とする。

## 無進捗と終了

対象に関係する正味の変更が0件なら、子のstatusにかかわらず最初の発生で全体ループを
停止する。親が修正、再試行、別タスクの開始を行わない。進められない理由がなければ、
その報告欠落も最終応答へ含める。

子が具体的なブロッカーや判断事項を `state.md` に記録した場合は変更として認める。
親はstateを再評価し、実行可能な別タスクがあれば継続する。すべてが判断待ちまたは
ブロック中になった時点でまとめて終了する。

同じtask IDを再委譲できるのは、前回の子がそのタスクの分解、ブロッカー、判断事項を
`state.md` に記録した場合だけとする。それ以外で同じtask IDが変更を伴いながら
未完了のまま残った場合は、無進捗と同様に全体ループを停止する。
