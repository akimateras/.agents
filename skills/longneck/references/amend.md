# Longneckタスク定義改訂契約

## 目次

- 目的と成功条件
- 対象と変更内容の確定
- `description.md` の改訂
- 検証と報告

## 目的と成功条件

ユーザーが明示した変更だけを `description.md` へ反映し、残タスクへの影響を報告する。
この契約は、ユーザーの明示指示を受けた `longneck-amend`、またはユーザーの変更指示を
受けて進行中の `longneck` 親が実行する。暗黙に実行しない。

成功条件は次のすべてとする。

- ユーザーが明示した変更だけが `description.md` へ反映されている。
- `task-format.md` のtemplate、構造、サイズ上限が守られている。
- 実際に `description.md` を変更した場合だけ、`progress.md` の作業ログへ
  `管理作業` の記録が1件追加されている。
- 対象内の変更が `description.md` と `progress.md` の管理作業ログだけである。
- 指示された変更が既に反映済みの場合は、更新日時と管理作業ログを含む
  一切の変更がない。
- 残タスクへの影響が報告されている。

## 対象と変更内容の確定

1. `targeting.md` の共通手順でLongneckルートを確定する。完全一致のtitleの明示指定を
   必須とし、自動選択しない。`longneck` 親が実行する場合は、`targeting.md` の
   引き継ぎ規則に従い、進行中の確定済み root と title を再解決しない。
2. 実行主体が `longneck-amend` の場合は、CLIの `lock` で対象titleのロックを取得し、
   終了時に自分のtokenで解放する。`longneck` 親が実行する場合は、親が保持する
   ロックの下で行い、新たに取得しない。
3. ユーザーの変更指示を、`description.md` のどのセクション（目的、スコープ、
   非スコープ、完了条件、長期制約、主な作業対象）のどの意味を変えるかへ落とし込む。
   変更内容を一意に特定できなければ、不足する一点を具体的に聞き返す。ユーザー指示に
   ない変更を混ぜない。
4. CLIの `validate` を成功させ、`manifest` の結果を基準としてリポジトリ外の
   一時ファイルへ保存する。基準を `.longneck` や作業ツリーへ置かない。
   `validate` のエラーがある対象は、対象タスクの文書が壊れているため改訂しない。
5. `description.md` と `state.md` の全文、`issues.md` の未解決一覧を読み、変更が
   残タスクへ与える影響を把握する。

## `description.md` の改訂

1. 指示された変更が既に `description.md` へ意味として反映済みかを先に確認する。
   反映済みで正味の変更がなければ、`description.md`、`progress.md`、更新日時を
   含めて何も変更せず、`no_change` として報告する。
2. `task-format.md` のtemplateと構造を維持し、ユーザーが明示した変更だけを適用する。
   サイズ上限（100行・8 KiB）を超えないよう、意味を失わずに簡潔へ保つ。
3. 現在の進捗、残タスク、作業履歴、詳細調査を `description.md` へ書き込まない。
4. 実際に `description.md` を変更した同じ作業単位で、`progress.md` の `作業ログ`
   先頭へ `管理作業` の記録を1件追加し、改訂の要旨と、根拠がユーザー指示である
   ことを残す。
5. `state.md`、`issues.md`、`docs/`、他titleを変更しない。改訂によりスコープ外へ
   出る残タスク、新しい完了条件に対して不足する作業などの不整合は、修正せず影響
   一覧として報告し、解消は `longneck` の進行またはユーザー判断へ委ねる。

## 検証と報告

CLIの `validate` を実行し、開始時に保存した `manifest` の結果を基準に、
`manifest-diff --baseline <保存したJSON> --allow description.md --allow progress.md`
で、対象内の変更が `description.md` と `progress.md` の管理作業ログだけである
（`no_change` では変更が0件である）ことを確認する。

次のいずれかの `status` で報告する。

- `amended`: 明示された変更を適用し、検証が成功した。
- `no_change`: 指示された変更が既に `description.md` へ反映済みで、正味の変更がない。
- `needs_user`: 変更内容を一意に特定できず、聞き返しが必要である。
- `failed`: 検証失敗、書込み失敗、または基準fingerprintとの不一致を検出した。

statusに加えて、改訂前後の差分の要旨（`no_change` では反映済みと判断した根拠）、
影響を受ける残タスクID、推奨する後続（例: `longneck` での再計画）を返す。
コミットの要否はユーザー指示とプロジェクト固有ルールに従う。
