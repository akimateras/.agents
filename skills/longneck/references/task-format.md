# Longneckタスク形式

この契約はLongneck文書の形式と分類を定める。スリム化の運用、Git管理の中立性、
同時変更の検出と排他は [operations.md](operations.md) に定める。

## 目次

- ルートとtitle
- 正本と参照方向
- ディレクトリ構造
- 文書別の契約
- `state.md` と `issues.md` の境界
- 機密情報と安全性
- 検証

## ルートとtitle

Gitリポジトリ内では `git rev-parse --show-toplevel` の結果をLongneckルートとする。
Gitリポジトリ外では、スキルを起動した時点の作業ディレクトリをルートとする。
途中でディレクトリを移動しても基準を変えない。bare repositoryでは作業を開始しない。

titleは `^[a-z0-9]+(?:-[a-z0-9]+)*$` に完全一致させる。先頭・末尾・連続ハイフン、
大文字、空白、記号、区切り文字を許可しない。ユーザー指定titleが不正なら無断で
正規化せず、有効な候補を提案して聞き返す。既存titleとの衝突では上書きしない。
エージェントが命名する場合は、目的を表す短いlowercase kebab-caseを作り、必要なら
意味のある語または数値suffixで衝突を避ける。

`.longneck` と `.longneck/<title>` がsymlink、通常ディレクトリ以外、または
Longneckルートの外へ解決される場合は停止する。

## 正本と参照方向

Longneck文書を、そのタスクの目的、実行状態、作業履歴、未解決知見の正本とする。
製品の挙動、公開契約、利用手順、設計仕様など、既にリポジトリ外側の適切な正本がある
事実はLongneckへ複製せず、Longneck側から恒久的な正本を参照する。

参照方向を次のように固定する。

- `.longneck` 内から、ソースコード、テスト、README、恒久的なdocsを参照してよい。
- `.longneck` 外のソースコード、コメント、テスト、文書、設定、コミットメッセージに、
  Longneckのパス、title、内部IDへの逆参照を作らない。
- 外へ知見を移すときは内容を自己完結させ、「Longneckを参照」のような逆参照を作らない。
- 別titleの文書を正本として参照しない。共有知識は `.longneck` 外の適切な正本へ置き、
  両方のタスクから外向きに参照する。

一般的な仕組みとして `.longneck` を説明するLongneckスキル自身の文書は、特定タスクへの
逆参照ではない。

## ディレクトリ構造

各タスクを次の構造にする。

```text
.longneck/
└── <title>/
    ├── description.md
    ├── state.md
    ├── progress.md
    ├── issues.md
    └── docs/
```

`.longneck` 直下には、各titleのディレクトリのほか、CLIが管理する助言的ロックの
`.locks/` と、作成中断の一時ディレクトリだけが存在してよい。
タスク直下へ他のファイルやディレクトリを追加しない。補助文書は `docs/` に置く。
`docs/` は作成時に空でよい。Gitは空ディレクトリを追跡しないため、追跡運用でも
checkout後に消えることがある。必要時はCLIの `repair-docs` で空の `docs/` を再作成し、
追跡のためだけの `.gitkeep` は自動作成しない。

CLIの `create` が中断されると、`.longneck` 直下に `.<title>.tmp-` で始まる一時
ディレクトリが残ることがあり、CLIの `list` は `stale_entries` として報告する。
これは未完成の作成物であり、タスクとして扱わない。削除にはCLIの `clean-stale` を
使い、手作業の `rm` で代用しない。実行中の `create` を誤って壊さないよう、CLIは
更新から1時間以上経過した一時ディレクトリだけを `stale_entries` として扱い、
それより新しいものは無視する。

すべてのMarkdownをUTF-8で記述し、末尾を改行する。履歴と現在状態を重複させない。
文書中の日時は、offsetと `T` 区切りを含むRFC 3339で書き、現在時刻は同梱CLIの
`now` で取得する。

## 文書別の契約

### `description.md`

長期間変わらない識別情報だけを書く。冒頭に `概要: <一文>` を置き、次を短く記述する。

- 目的
- スコープ
- 非スコープ
- 全体の完了条件
- 長期間有効な制約
- 主な作業対象パス

現在の進捗、残タスク、作業履歴、詳細調査を入れない。100行または8 KiBのいずれかを
超えると検証warningとして扱う。ユーザーが目的・スコープ・完了条件を変更した場合だけ、
[amend.md](amend.md) の契約に従って更新し、子が都合に合わせて変更しない。

```markdown
# <title> — タスク定義

概要: <一文>

## 目的

<達成する結果>

## スコープ

- <対象>

## 非スコープ

- <対象外>

## 完了条件

- <観測可能な完了条件>

## 長期制約

- <制約>

## 主な作業対象

- `<path>`
```

### `state.md`

現在の状態だけを書く。冒頭にRFC 3339の `更新日時` と、次のいずれかの `状態` を置く。

- `active`: `調査待ち` または `実行待ち` に残タスクが1件以上ある。依存関係で
  当面着手できないタスクだけでも `active` とする。
- `awaiting-user`: `調査待ち` と `実行待ち` が空で、`判断待ち` に残タスクがある。
  `ブロック中` と併存してよい。
- `blocked`: 残タスクが `ブロック中` だけにある。
- `completed`: 完了条件を満たし、4区分すべてに残タスクがない。

CLIの `validate` は、この区分別件数から `状態` を機械判定し、不一致をエラーとして
報告する。`現在のスナップショット` と4種類の残タスクセクションを持つ。
完了済みタスクや変更履歴は残さない。

`現在のスナップショット` は、現在の作業段階、直近の焦点、タスク選択に必要な事実へ
限定する。製品挙動、設計仕様、利用手順などの恒久的な正本が外部にある場合は、詳細を
複製せず正確なpathを参照する。

タスクにはtitle内で一意な `T-0001` 形式（数字4桁、10000件以降は桁を増やす）の
安定IDを付ける。見出しは `### T-0001 — <名前>` 形式とし、内容を表す短い名前を
必ず付ける。名前のない見出しはCLIの `validate` がwarningとして報告する。
`調査待ち` と `実行待ち` のタスクに `- 目的:` または `- 完了条件:` の行がない
場合も、CLIの `validate` と `tasks` はwarningとして報告する。
移動・分解まで同一作業ならIDを維持し、別の完了条件へ分解した子タスクには
新しいIDを付ける。削除したIDを再利用しない。完了タスクは `state.md` から除かれ、
使用済みIDは作業ログだけに残るため、`state.md` だけを見て採番しない。新しい
タスクIDの採番には、`state.md`、`progress.md`、`docs/work-log-archive.md`
（存在する場合）の全T-IDより大きい次のIDを返すCLIの `next-task-id` を使う。

`依存関係` には `なし`、現存する残タスクのID列（空白・読点・カンマ区切り）、または
IDで表せない前提条件の短い記述を書く。1つの `依存関係` 行でID列と自由記述を混在
させない。ID依存と自由記述の前提が併存する場合は、タスクを分解するか、前提全体を
自由記述として書く。参照先タスクの完了により満たされた依存は、完了タスクを
`state.md` から除く同じ作業単位で `依存関係` からも取り除く。CLIの `validate` は、
存在しないIDへの参照をwarning（完了済みとして満たされたとみなす）、自己依存と依存の
循環をエラーとして報告する。残タスク、依存グラフ、実行可能なタスクIDの機械可読な
一覧には、CLIの `tasks --root <root> --title <title>` を使う。`tasks` が `runnable`
を機械判定するのは `なし` と純粋なID列だけであり、自由記述、またはIDと自由記述の
混在（warningとして報告する）は `runnable` を `null` にして呼出元の判定に委ねる。
`tasks` は各タスクへ、そのタスクの完了により依存が満たされる後続IDの一覧を
`unlocks` として、見出しから本文末尾までの原文を `body` として報告する。
子への委譲などでタスク項目の全文が必要な場合は、`state.md` からの手動転記では
なくこの `body` を使う。

```markdown
# <title> — 現在の状態

更新日時: <RFC3339>
状態: active

## 現在のスナップショット

<現在成立している事実と直近の焦点>

## 調査待ち

### T-0001 — <名前>

- 目的: <調べて明らかにすること>
- 完了条件: <分解または実現性判断ができる状態>
- 依存関係: なし

## 判断待ち

なし。

## 実行待ち

なし。

## ブロック中

なし。
```

`判断待ち` にはユーザーへ判断してほしい問いを明記し、複数案にはpros/consを付ける。
`実行待ち` は一度に検証できる作業単位にする。`ブロック中` にはブロッカー、解除主体、
観測可能な再開条件を書く。

### `progress.md`

設計判断と作業履歴を書く。`設計判断` の各項目は、次の形式の第3レベル見出しで記録する。
CLIの `validate` は、この `### D-0001` 見出し形式を前提にIDの重複を検査するため、
IDを箇条書きや本文の中だけに書かない。

```markdown
### D-0001 — <決定の短い名前>

- 決定: <採用した内容>
- 理由: <根拠>
- 影響: <波及範囲と再評価の観点>
```

後から覆す場合も削除せず、新しい決定から旧IDを明示して経緯を残す。
`longneck-flush` が外部正本への収束後に設計判断を除いた場合は、管理作業ログに記録された
退役IDも使用済みとみなし、再利用しない。新しいIDの採番には、`progress.md` と
`docs/work-log-archive.md`（存在する場合）の全D-ID（退役IDと範囲の端点を含む）より
大きい次のIDを返すCLIの `next-decision-id` を使う。

`作業ログ` を必ず最後のセクションにし、新しい記録を先頭へ追加する。各記録にRFC 3339、
対象タスクID、実施内容、検証結果を含める。Longneck文書自体の管理作業だけは、対象タスクIDの
代わりに `管理作業` と明記する。作業ログは純粋な時系列記録とし、後続作業に必要な知見を
埋め込まない。知見は同じ作業単位内で `state.md`、`issues.md`、`docs/`、または恒久的な
外部正本へも分類する。

```markdown
# <title> — 進捗

## 設計判断

なし。

## 作業ログ

### <RFC3339> — T-0001

- 実施: <内容>
- 検証: <コマンドまたは確認と結果>
```

### `issues.md`

現在の目的には不要だが将来無視できない未解決知見だけを書く。各項目は次の形式の
第3レベル見出しで記録する。CLIの `validate` は、この `### I-0001` 見出し形式を
前提にIDの重複を検査するため、IDを箇条書きや本文の中だけに書かない。
単なる未実装一覧や現在必要な作業を入れない。削除したIDを再利用しない。
新しいissue IDの採番には、`issues.md`、`progress.md`、`docs/work-log-archive.md`
（存在する場合）の全I-IDより大きい次のIDを返すCLIの `next-issue-id` を使う。

```markdown
### I-0001 — <名前>

- 発見: <内容>
- 影響: <将来の安全性・品質・判断への影響>
- 証拠: <再現手順または根拠>
- 再検討条件: <再評価する時期または観測可能な条件>
```

issueが現在の目的に必要または実行対象になったら、詳細を `state.md` へ移し、
`issues.md` から削除する。移動の履歴を `progress.md` に残し、両方で保守しない。
解消・無効化したissueも `issues.md` から削除し、必要な経緯を `progress.md` または
恒久的な外部正本へ残す。`progress.md` へ残す移動・解消の経緯には対象のI-IDを
明記し、削除済みIDの再利用を防ぐ根拠を作業ログへ残す。

```markdown
# <title> — 将来の検討事項

現在の目的の達成には不要だが、将来の判断に必要な未解決知見を記録する。
現在の完了条件に必要な作業や判断は `state.md` に置く。

## 未解決

なし。
```

### `docs/`

詳細調査、比較、実験結果など、再利用する任意の永続的な作業記憶を置く。
通常のUTF-8テキスト文書だけを使い、symlinkとバイナリ成果物を置かない。
秘密値は形式を問わず記録しない。
すべての文書を次の3行で始め、本文を変更したときだけ更新日時を変える。

```text
作成日時: <RFC3339>
更新日時: <RFC3339>
概要: <この文書が何であるかを表す一文>
```

この3行は `docs/` 配下のすべてのファイルに必須であり、JSONやCSVなどの生データ
単体は置けない。再利用する生データは、この3行で始まるMarkdown文書へ埋め込むか、
適切な外部正本へ置く。

`docs/work-log-archive.md` は、`longneck-flush` が `progress.md` から退避した古い
作業ログの置き場として予約する。このファイルの作成と変更は
[flush.md](flush.md) の契約だけが行い、他の用途に使わない。CLIの `validate` は、
このファイルについて3行メタデータに加え、イベント見出し
（`### <RFC3339> — <task IDまたは管理作業>`）の形式と新しい順の並びも検査する。

## `state.md` と `issues.md` の境界

`description.md` の未達成の完了条件を満たすために、解決、実行、調査、ユーザー判断が
必要なものは `state.md` に置く。現在合意された目的とスコープの達成には不要で、
現時点で実行を予定しないが、将来の安全性、品質、保守性、判断に影響する知見だけを
`issues.md` に置く。

高コスト、困難、時間不足、面倒であることを、必要なタスクを `issues.md` へ移す理由に
しない。現在の正しさや完了可否が不明なら `state.md` の `調査待ち` に置く。
現在の目的に必要なユーザー判断は `判断待ち`、外部状態の変化が必要なら `ブロック中` に
置く。重大なセキュリティ、プライバシー、データ損失の懸念は、スコープ外でも黙って
延期せずユーザーへ通知する。

## 機密情報と安全性

資格情報、token、秘密鍵、個人情報、顧客実データ、秘密値をLongneck文書へ記録しない。
発見した場合は値を表示または複製せず、保存場所だけを報告し、適用される安全手順に従う。
永続化でも機密値を別文書へ転記しない。

既存変更を保持し、対象外ファイルを巻き戻さない。未知のファイル、壊れた形式、symlinkを
推測で修正しない。2文書のスリム化は `longneck-flush`、`description.md` の改訂は
[amend.md](amend.md) の契約（`longneck-amend` または `longneck` 親）、形式の修復は
[repair.md](repair.md) の契約（`longneck-repair`）、タスク削除は
`longneck-sever` だけが、それぞれの明示契約に従って行う。`判断待ち` と `ブロック中` の
解除は、[worker-contract.md](worker-contract.md) の「ユーザー判断とブロッカー解除」の
契約に従い、`longneck` 親だけが行う。

## 検証

同梱CLIで次を確認する。

```text
python3 <longneck-skill>/scripts/longneckctl.py validate --root <root> --title <title>
python3 <longneck-skill>/scripts/longneckctl.py fingerprint --root <root> --title <title>
```

CLIのexit codeは共通して、`0` が成功（該当なし）、`1` が構造化出力に該当あり
（`validate`・`list`・`tasks` の形式エラー、`lock` の競合、`manifest-diff` の許可外
変更、`backlinks` の検出）、`2` が前提違反などで実行不能（標準エラー出力に
`longneckctl:` のメッセージ）を表す。`1` でも標準出力のJSONは有効であり、コマンド
失敗と扱わず内容を読んで対応を判断する。例外として `backlinks` は、走査エラーが
あると部分的な走査結果を含む有効なJSONを標準出力へ出したまま、標準エラー出力の
`longneckctl:` メッセージとともに `2` を返す。この場合、走査は不完全であり、
参照が存在しないことの根拠にしない。

`validate` のエラーは作業開始前に解決する。`description.md` のサイズ警告は修正せず
報告し、整理が必要ならユーザーの明示指示を得て [amend.md](amend.md) の契約で行う。
ユーザーの手編集を勝手に捨てない。Longneckスキル群は
壊れた形式を暗黙に修復しない。明示的に許可された例外（CLIの `repair-docs` による、
checkoutで消えた空の `docs/` の再作成と、CLIの `clean-stale` による
`stale_entries` の削除）を除き、形式の修復はユーザーの手作業、または明示起動された
`longneck-repair` の [repair.md](repair.md) に基づく承認付き契約で行う。
`longneck-persist` と
`longneck-sever` は、壊れた形式からの知見救出を妨げないため、通常ファイルだけで構成された
安全なタスクツリーなら形式上のエラーも内部一覧へ含めて処理する。
