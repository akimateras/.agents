# Longneckオーケストレーション契約

## 目次

- 適用範囲
- 対象を選ぶ
- ロックと基準状態
- ユーザー判断とブロッカー解除
- 一作業単位ずつ進める
- 子の終了を静かに待つ
- 終了を判定する
- 最終報告を返す

## 適用範囲

この契約は `longneck` 親だけが実行する。親は選択したタスクの全体管理、子への委譲、
成果の検証、終了判定を行い、自身は一作業単位の実装担当にならない。
同じtitleに対して複数の親を並行実行しない。

## 対象を選ぶ

1. [targeting.md](targeting.md) の共通手順でLongneckルートを確定する。
2. ルートに適用される `AGENTS.md` を読み、プロジェクト固有の文書・作業領域の
   分岐を確認する。
3. CLIの `list --root <root>` で全titleを検査する。壊れたタスクを推測で補修せず、
   [task-format.md](task-format.md) が明示的に許可する2つの例外（CLIの
   `repair-docs` と `clean-stale`）だけをCLIで行う。title非依存の `clean-stale`
   はこの段階で行ってよいが、`repair-docs` は対象titleへの書込みなので、対象の
   確定とロック取得の後に行う。その他の形式エラーの修復は、
   ユーザーの手作業または明示起動された `longneck-repair` に委ねる。
4. titleが0件なら、対象がないことと `longneck-spawn` が必要なことを返す。
5. titleの選択と、唯一titleの自動選択の可否は `targeting.md` の
   「スキル別の差分」に従う。
6. ユーザーが目的・スコープ・完了条件の変更を明示した場合は、子へ委譲せず
   [amend.md](amend.md) の契約で `description.md` を先に改訂し、残タスクへの
   影響を確認してから進行する。
7. 選択後にCLIの `validate --root <root> --title <title>` を通し、
   `description.md` と `state.md` の全文と `issues.md` を読む。`progress.md` は
   設計判断、直近の作業ログ、選択タスクに関係する記録を読む。`docs/` は一覧を
   確認し、選択タスクに関係する文書だけを読む。
8. `description.md` の作業対象パスから、適用される入れ子の `AGENTS.md` と
   恒久的な正本を特定して読む。

## ロックと基準状態

対象の確定後、最初の書込みより前に、CLIの `lock` で対象titleの助言的ロックを
取得する。取得できなければ、[operations.md](operations.md) の
「同時変更の検出と排他」に従い、保持者情報を報告して停止する。ロックは
最終報告の前に自分のtokenで解放する。

実行前に次を記録する。

- Gitリポジトリなら `HEAD`、branch、`git status --short`、staged/unstaged差分
- CLIの `fingerprint --root <root> --title <title>` の結果
- この実行中に完了した作業単位と検証結果
- ユーザー判断待ち、外部ブロッカー、記録不能な知見

`.longneck` が未追跡でも更新を検出できるよう、Git差分だけで判定しない。
実行開始前から作業ツリーに存在した変更を子の成果とみなさず、上書きまたは
巻き戻しを行わない。

## ユーザー判断とブロッカー解除

`判断待ち` と `ブロック中` の解除は親だけが行い、子へ委譲しない。ループの開始前と、
`state.md` を読み直すたびに、[worker-contract.md](worker-contract.md) の
「ユーザー判断とブロッカー解除」の契約を適用する。

## 一作業単位ずつ進める

終了条件を満たすまで次を繰り返す。

1. `state.md` を読み直す。
2. CLIの `tasks --root <root> --title <title>` で残タスクと依存グラフを列挙し、
   依存関係を満たす `調査待ち` と `実行待ち` を確定する。CLIが `runnable` を
   判定できないタスク（自由記述、またはIDと自由記述が混在した `依存関係` を持ち、
   `runnable` が `null` のもの）は、`state.md` の記述から自分で判定する。
   `判断待ち` と `ブロック中` は実行しない。
3. プロジェクト固有の優先順に従う。優先順がなければ、CLIの `tasks` が各タスクへ
   報告する `unlocks`（完了により依存が満たされる後続ID）を比較し、後続を最も多く
   解放するタスクを選ぶ。同数なら最小のtask IDを選ぶ。大きすぎるものは
   調査・分解だけを一単位にする。
4. 起動直前のGit状態とLongneck fingerprintを記録する。
5. 同じモデルを継承する新しい子を1つだけ起動し、
   [worker-contract.md](worker-contract.md) のプロンプト契約、完全一致の
   root/title/task ID、完了条件を含む対象タスク全文（CLIの `tasks` が報告する
   該当タスクの `body`）、コミット方針を渡して
   `$longneck-head` を明示的に使わせる。
6. 子の終了まで待つ。同時に別の子を起動せず、子に次タスクを選ばせない。
7. 子の報告を、`worker-contract.md` の「親による照合」に従い、Git差分、
   Longneck fingerprint、変更ファイル、検証結果と照合する。
8. 新しい永続的知見が適切なファイルへ記録済みか確認する。通常終了時に親の
   コンテキストだけへ知見を残さない。安全に補記できる場合だけ親が補い、根拠不足や
   書込み不能なら、終了判定の例外的中断として全体ループを停止する。
9. 対象タスクに関係する正味の変更の有無を実ファイルで判定する。無進捗の扱いは
   `worker-contract.md` の「無進捗と終了」に従う。
10. 変更があれば `state.md` を再読し、実行可能タスクを再計算する。子が報告した
    「次のタスク」をそのまま採用しない。同じtask IDの再委譲の可否も
    「無進捗と終了」に従う。

## 子の終了を静かに待つ

利用中のハーネスの完了待ち機構で子の終了を待つ。待機のタイムアウトを指定できる
場合は許される最大値を指定し、便宜上短くしない。

- 子の実行中に待機がタイムアウトした場合は、同じ条件ですぐに待機を再開する。
  タイムアウトを停滞、失敗、進捗報告の契機とみなさず、それだけを理由にユーザーへの
  報告、子への連絡・催促・中断、別の子の起動、次タスクの選定、作業ツリーの変更を
  行わない。上位指示が待機中の定期的なユーザー向け更新を必須にする場合だけ、
  その頻度で最短の現況を通知し、その場合も子へは連絡しない。
- 子からの途中メッセージが単なる進捗報告なら、返信せず待機を続ける。子が明示的に
  情報を求めた場合は、ユーザー指示または正本で確定済みの事実だけを短く返し、助言、
  範囲の変更、品質を下げた早期完了、次の作業への移行を提案しない。
- ユーザーによる中止または指示変更、確認済みの安全上の問題がない限り、実行中の
  子を中断しない。

## 終了を判定する

次のいずれかで終了する。

- `state.md` の状態が `completed` で、残タスクがない。
- 実行可能な `調査待ち`・`実行待ち` がなく、残タスクがすべて `判断待ち` または
  `ブロック中`（混在を含む）である。
- `worker-contract.md` の「無進捗と終了」が全体ループの停止を求めた。
- 権限、ツール障害、壊れた正本、記録できない永続的知見などにより安全に
  継続できない（例外的中断）。

`state.md` が `completed` になった場合は、終了前に `description.md` の完了条件と
照合する。未達の完了条件があれば完了として扱わず、不整合として停止し、最終報告で
指摘する。

実行可能な `調査待ち` または `実行待ち` が残り、無進捗も例外的中断もなければ
継続する。時間や難しさだけで終了しない。

## 最終報告を返す

終了前に、取得したロックを自分のtokenで `unlock` する。解放できない場合は
その旨を報告へ含める。次を簡潔に返す。

- 完了、全残件が判断待ち・ブロック中、無進捗、例外的中断のいずれか
- この実行で完了した作業単位と主要な変更
- 検証と結果
- ユーザー判断事項または再開条件
- Longneck文書や成果物へ記録できなかった知見。なければその旨

タスク完了、文書量、形式エラーを理由に `longneck-persist`、`longneck-flush`、
`longneck-sever`、`longneck-repair` を自動実行しない。
