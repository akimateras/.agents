## 目的と成功条件

長期間の作業で肥大化した `state.md` と `progress.md` から、現在のタスク管理または
純粋な作業履歴に不要な情報を除く。除去する技術的・業務的な意味は、先に `.longneck`
外の既存正本へ収束させる。

成功条件は次のすべてとする。

- 除去した知見が適切な外部正本で自己完結している。
- `state.md` に現在のタスクを選択、実行、再開するための情報が残っている。
- `progress.md` と、退避した場合は `docs/work-log-archive.md` に時系列の作業事実が
  残り、恒久知識が混在していない。
- 残タスクのID、区分、意味、依存関係、全体状態を変更していない。
- `description.md`、`issues.md`、`docs/work-log-archive.md` を除く `docs/`、
  他titleを変更していない。
- Longneck形式の検証と、変更先に関係する検証が成功している。
- 同じ入力で再実行しても、時刻更新や管理ログ追加を含む変更が発生しない。

`longneck-flush` の成功は `.longneck/<title>` 全体の削除可能性を保証しない。削除前には
`longneck-sever` が `full-task` 範囲の永続化を別途実行する。

## 対象範囲

完全一致の1つのtitleについて、内容を変更してよいのは `state.md`、`progress.md`、
および「作業ログのアーカイブ」に従う `docs/work-log-archive.md` だけとする。
`description.md`、`issues.md`、その他の `docs/` 文書は意味の判定に必要な文脈として
読んでよいが、整理、修復、追記、削除を行わない。

次は `longneck-flush` の範囲外とする。

- タスクの実行、完了、追加、分解、並べ替え、区分変更
- ユーザー判断の解決、ブロッカーの解除、全体状態の変更
- `description.md` の目的、スコープ、完了条件、長期制約の改訂
- `issues.md` の登録、解消、移動、圧縮
- 作業ログのアーカイブを除く、`docs/` 文書の統合、要約、削除
- 製品挙動、公開契約、実装の修正
- Longneckタスクの削除

`validate` のエラーがある対象はスリム化しない。`longneck-persist` と `longneck-sever` が
壊れた形式から知見を救出できることと、内容を除去する `longneck-flush` の開始条件を
区別する。

## 情報の分類

`state.md` と `progress.md` の全内容を、見出しではなく意味単位で次へ分類する。

### `state-management-only`

Longneck内で現在のタスク状態を管理するために必要で、`longneck-flush` 後も残す情報とする。

- title、更新日時、全体状態
- 現在の作業段階、直近の焦点、タスク選択に必要な短いスナップショット
- 残タスクの安定ID、区分、名前、目的、完了条件、依存関係
- 判断待ちの問い、選択肢、利点・欠点、判断後に解放されるタスク
- ブロッカー、解除主体、観測可能な再開条件
- 上記を自己完結させるため必要な、外部正本またはLongneck内の補助文書への参照

管理に必要という理由だけで、完了済み機能一覧、過去のフェーズ一覧、実装詳細、設計理由、
調査結果、古い検証件数を残さない。長い説明が外部正本にある場合は、同じ意味を複製せず
参照へ置き換える。

### `work-log-only`

将来の仕様判断に使う恒久知識ではなく、いつ、何を行い、どの検証がどう終わったかを示す
時系列記録とする。

- RFC 3339の実施時刻
- 対象タスクIDまたはLongneck文書の管理作業であること
- 実施した変更、調査、判断依頼の事実を表す短い要約
- 実行した検証と、その時点の成功・失敗・未実施の結果
- ローカルコミットなど、履歴を特定するための識別情報
- 過去の管理作業記録が列挙した退役設計判断ID。ID再利用防止の唯一の根拠であり、
  全量を維持する

作業ログに混在する仕様、判断と理由、再利用できる調査結果、外部契約、制約、残件、
再開条件は `work-log-only` ではない。これらを外部へ永続化してからログから除く。

### `removable-knowledge`

上記2分類に該当せず、削除後も利用者または将来のエージェントが必要とする意味とする。
主な対象は次のとおりとする。

- 完了済み機能と現在成立している製品挙動の詳細
- 設計判断、却下案、理由、影響、再評価条件
- 外部契約、互換性、運用、安全性、移行に関する制約
- 再現可能な調査結果、検証済みの前提、重要な証拠
- 履歴へ混入した未解決事項、残作業、ブロッカー、再開条件
- 外部正本の代わりにLongneckへ置かれている利用手順または設計仕様

見出し、空の節、重複、古い時刻、内部ID自体、再現価値のない診断ノイズは知見ではない。
古い記録が後の記録で覆されている場合は、現在の決定と再評価に必要な理由だけを外部へ
反映し、無効な案を現行仕様として書かない。

## `longneck-persist` との境界

`longneck-flush` は、ファイルを変更する前に縮約後の2文書をメモリ上で組み立てる。
変更前には存在し、候補からは消える意味を全件特定したうえで、変更前の対象へ
`longneck-persist` を `flush-state-progress` 範囲で実行する。この範囲の永続化対象は
候補から除く `removable-knowledge` だけとし、候補に残す `state-management-only` と
`work-log-only` は外部へ複製しない。

永続化後に候補を変え、新しい意味を除去対象へ加えてはならない。候補を変える必要が
生じた場合は、その意味を内部一覧へ加え、`longneck-persist` の調査からやり直す。

`description.md`、`issues.md`、`docs/` は次の目的に限って参照する。

- 選択した知見の意味、現在性、出典を確かめる。
- 同じ意味が複数のLongneck文書にあることを検出する。
- 外部正本との矛盾を見つける。
- `longneck-flush` 後も必要な内部参照を確定する。

これらの文書だけに存在する知見を内部一覧、ギャップ、外部変更へ含めない。選択した
知見がこれらにも重複しているだけでは、`.longneck` 外へ永続化済みとはみなさない。
この限定により、`issues.md` や `docs/` 固有の情報を副次的に外部へ複製して二重管理を
作らない。

スリム化へ進めるのは、`longneck-persist` の終了報告が次の受入条件をすべて満たす
場合だけとする。

- `status` が `persisted` または `already_persisted` である。
- `source_scope` が `flush-state-progress` である。`full-task` の報告をスリム化成功の
  根拠へ読み替えず、必ず `flush-state-progress` の報告を得る。
- 未解決ギャップが0件で、関係する検証が成功している。
- 報告された `source_fingerprint` が現在値と一致している。
- `deletion_ready: false` である。

満たさない場合はLongneck文書を変更しない。

## `state.md` の縮約

縮約後は `task-format.md` の標準構造だけを使う。

1. title見出し、更新日時、状態を残す。実際に内容を変更した場合だけ更新日時を変える。
2. `現在のスナップショット` を、現在の作業段階、直近の焦点、タスク選択に必要な事実と
   正確な参照だけへ短縮する。
3. `調査待ち`、`判断待ち`、`実行待ち`、`ブロック中` をこの順序で1つずつ残す。
4. 各残タスクのID、区分、名前、目的、完了条件、依存関係と、その区分に必要な詳細を
   意味を変えず残す。
5. 独自の第2レベル見出し、完了済み一覧、履歴、設計説明、提供機能一覧、詳細な検証結果、
   参照資料一覧を除く。現在の残タスクに必要な個別参照だけをスナップショットまたは
   タスク項目へ残す。

タスク記述が長いことだけを理由に完了条件、判断材料、再開条件を省かない。内容が誤って
いる、区分が不整合、タスクが大きすぎるなど、スリム化以外の修正が必要だと判明した場合は
候補へ混ぜず、変更前に停止して別作業として報告する。

## `progress.md` の縮約

title見出し、空の `設計判断`、最後の `作業ログ` だけを第2レベル見出しとして残す。

1. `設計判断` の全知見が外部正本へ収束したことを確認し、本文を `なし。` にする。
2. 既存の各作業イベントを削除せず、新しい順序、時刻、対象タスクIDを維持する。
   古いイベントの退避は、この縮約とは別に「作業ログのアーカイブ」の契約だけで行う。
3. 各イベントを、短い `実施` と `検証`、必要な履歴識別情報だけへ縮約する。過去の
   管理作業記録が列挙した退役設計判断IDは履歴識別情報として全量維持し、縮約、
   範囲統合の変更、省略を行わない。
4. 実装仕様、判断理由、詳細調査、未解決事項、後続計画、現在状態をログから除く。
5. 検証の成功・失敗を反転させず、記録にない成功やコマンドを補完しない。

実際に内容を除去した場合は、`作業ログ` の先頭に次の管理作業記録を1件だけ追加する。

```markdown
### <RFC3339> — 管理作業

- 実施: `state.md` と `progress.md` の除去予定知見を外部正本へ永続化し、2文書をスリム化した。退役設計判断ID: <全IDまたはなし>。アーカイブしたイベント: <件数またはなし>
- 検証: <Longneck形式、変更範囲、外部正本の検証結果>
```

退役させた設計判断IDは省略せず、連続して使用済みのIDだけを範囲でまとめ、それ以外は
個別IDで全件を表す。新しい設計判断には、`progress.md` と `docs/work-log-archive.md` の
管理作業記録に現れる最大値より大きい未使用IDを付ける。過去のIDを再利用しない。
未使用IDの確認にはCLIの `next-decision-id` を使う。既にスリム化済みで正味の変更が
ない場合は、更新日時も管理作業記録も追加しない。

## 作業ログのアーカイブ

縮約後も `progress.md` が [operations.md](operations.md) の「定期的なスリム化」の
しきい値を超える場合だけ、古い側から連続する作業イベントを
`docs/work-log-archive.md` へ退避してよい。イベントは純粋な作業記録のままLongneck内を
移動するだけなので、この退避に追加の永続化は不要である。

1. 少なくとも直近20件のイベントと、今回追加する管理作業記録は `progress.md` に残す。
2. アーカイブは `docs/` の3行メタデータ（作成日時、更新日時、概要）で始め、本文は
   `progress.md` の `作業ログ` と同じ `### <RFC3339> — <task IDまたは管理作業>` 形式の
   イベントを新しい順に置く。既存アーカイブへ追記するときは、今回退避するイベントを
   既存イベントより前（新しい側）へ挿入し、ファイル全体を新しい順に保つ。
3. イベントの内容、順序、時刻、対象タスクIDを変えずに移動する。退役設計判断IDを
   列挙した管理作業記録も移動してよい。CLIの `next-task-id`・`next-decision-id`・
   `next-issue-id` はこのアーカイブも読むため、ID再利用防止は移動後も保たれる。
4. 実際に変更したときだけアーカイブの `更新日時` を変える。新規作成時は
   `作成日時` と `更新日時` を同じ値にする。
5. しきい値を超えていない場合、または退避してもしきい値内へ収まる見込みの改善が
   ない場合は、アーカイブを作成・変更しない。

## 適用と復旧

CLIの `manifest` は対象全体の `fingerprint` と、相対パス順の `entries` を返す。
各項目に `path` と `type` があり、通常ファイルには `bytes`、`lines`、`sha256` もある。
開始時に、CLIの `lock` で対象titleのロックを取得し（解放は終了報告の前に自分の
tokenで行う）、`validate` の成功、`manifest` の結果、Git状態、CLIの `list` で
確定した他titleと各 `fingerprint` を基準として記録する。以後の変更範囲の確認には、
保存した `manifest` の結果を基準に渡すCLIの `manifest-diff` を使う。基準の
`manifest` JSONはリポジトリ外の一時ファイルへ保存し、`.longneck` や作業ツリーへ
置かない。

`longneck-persist` の成功後、適用直前に開始時の基準を渡した `manifest-diff`
（`--allow` なし）で変更が0件であることを確認する。変更があれば、候補を適用せず
新しい内容で `longneck-persist` からやり直す。

開始時の2文書と、存在する場合は `docs/work-log-archive.md` をメモリ上に保持し、
Longneck直下へバックアップや一時ファイルを作らない。`state.md`、`progress.md`、
必要ならアーカイブを同じ作業単位で変更した後、直ちに検証する。候補適用後の
検証に失敗した場合は、次のすべてを満たすときだけ変更したファイルを開始時の内容へ戻す
（開始時に存在しなかったアーカイブは削除する）。

- 現在の変更対象ファイルが自分の適用した候補と一致する。
- 他プロセスまたはユーザーによる同時変更がない。
- 戻す対象が `state.md`、`progress.md`、`docs/work-log-archive.md` だけである。

外部正本への正しい永続化は安全な追加成果なので巻き戻さない。既存のユーザー変更を
戻すために `git restore`、`git checkout`、`git reset` を使わない。

## 収束性と検証

次をすべて確認する。

- CLIの `validate` が成功する（アーカイブの3行メタデータとイベント見出しの
  形式・順序も検証される）。
- 開始時の基準と `--allow state.md --allow progress.md
  --allow docs/work-log-archive.md` を渡したCLIの `manifest-diff` が、
  これら以外の変更を報告しない。
- CLIの `list` で確定した他titleの一覧と `fingerprint` が開始時と終了時で一致する。
- 4種類の残タスクについて、ID、区分、意味、依存関係、件数が前後で一致する。
- 全体状態が前後で一致し、件数と整合する。
- 除去した全知見が外部正本で `represented` になっている。
- `progress.md` と、退避した場合はアーカイブの各イベントが純粋な作業記録だけに
  なっている。退避したイベントが内容を変えずに全件アーカイブへ存在する。
- 外部正本にLongneckのパス、title、内部IDへの逆参照がない。
- 外部変更の文書検査、テスト、lint、リンク検査などが成功している。
- 同じ分類を再実行して `removable-knowledge` が0件であり、候補差分も0件になる。

Git差分は補助として使い、Gitで未追跡または無視されているLongneck文書が不変であることは
ハッシュで確かめる。検証失敗、書込み失敗、権限不足、同時変更、未解決ギャップがあれば
成功を返さない。

## 終了報告

次の形で返す。

```text
status: flushed | already_slim | needs_user | conflict | failed
title: <exact-title>
persist_status: persisted | already_persisted | needs_user | conflict | failed
persist_source_scope: flush-state-progress
deletion_ready: false
source_fingerprint: <longneck-persistが検証したsha256>
result_fingerprint: <スリム化後のsha256、またはnone>
removed_knowledge_items: <count>
archived_events: <count、またはnone>
retired_decision_ids:
  - <ID範囲、個別ID、またはnone>
preserved_task_ids:
  - <ID、またはnone>
changed_external_files:
  - <path、またはnone>
changed_longneck_files:
  - <state.md / progress.md / docs/work-log-archive.md、またはnone>
size_change:
  - state.md: <before lines/bytes> -> <after lines/bytes>
  - progress.md: <before lines/bytes> -> <after lines/bytes>
untouched_longneck_sources:
  - description.md
  - issues.md
  - docs/（work-log-archive.md を除く）
  - other titles
unresolved_gaps:
  - <内容と理由、またはnone>
verification:
  - <検証と結果>
```

`flushed` と `already_slim` は `longneck-persist` の成功、未解決ギャップ0件、検証成功の
場合だけ使う。正味のLongneck変更が0件なら `already_slim` を使う。終了報告を独自の
監査ファイルとして保存しない。
