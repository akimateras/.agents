# 完了した作業単位をコミットする

## 目的

作業中のエージェントが論理的な区切りへ到達するたびに、完了した変更だけを新しい
ローカルコミットとして記録し、引き続き本来の作業を進める。このスキルの暗黙呼び出しを
ローカルでのステージとコミットに対する事前承認として扱い、実行前にユーザーへ確認しない。

コミット操作だけを担当し、メッセージの規則は `write-commit-messages` に一元化する。

## 必須の依存スキルを読み込む

コミットメッセージを作成またはレビューする前に、利用環境のスキル呼び出し方法で
`write-commit-messages` を読み込み、その指示に従う。スキルを名前で呼び出せない環境では、
このファイルを基準に `../write-commit-messages/SKILL.md` を読む。

依存スキルにはメッセージの作成とレビューだけを委譲する。ステージとコミットの操作は、
このスキルの手順で行う。依存スキルを利用できない場合は、独自の規則で代替せず、
コミットを延期して不足している依存関係を報告する。

## コミット可能な作業境界を判定する

次の条件をすべて満たす作業単位だけをコミットする。

- 1 つの機能、修正、リファクタリング、テスト整備、文書更新など、独立して説明し、
  取り消せる目的が完了している。
- その単位に必要な実装と付随するテストや文書が揃い、追加変更を予定していない。
- 実行可能な関連検証が成功している。該当する検証がない場合や安全に実行できない場合は、
  その事実を作業記録に残し、他の条件から完了を判断する。
- ユーザーの依頼、作業開始時の状態、エージェントが実際に行った編集から、対象差分を
  エージェント自身の作業へ帰属できる。
- コミット後のリポジトリが、後続作業の基点として一貫した状態になる。

独立した次の作業へ移る前、完了した実装を引き渡す前、または依頼全体を終える前に、
この条件を満たす単位があればコミットする。探索中の変更、仮のデバッグコード、失敗中の
検証に関係する変更、後続の編集がなければ成立しない中間状態はコミットしない。

条件を満たさない場合は、ユーザーへコミット確認を求めない。安全に続行できる作業があれば
元のワークフローへ戻り、次の境界で再評価する。対象の帰属、インデックス、機密情報などの
問題を自動で解消できなければコミットを延期し、通常の進捗報告または最終報告に理由を残す。

## 権限と安全条件

- このスキルが明示または暗黙に呼び出された時点で、条件を満たす対象への `git add` と
  新規 `git commit` は承認済みとして扱う。コミット計画やメッセージの確認をユーザーへ
  求めない。ただし、現在の依頼またはリポジトリの指示がコミットを禁止している場合は
  実行しない。
- エージェントが現在の作業単位で作成した変更、またはユーザーが明示した変更だけを扱う。
  作業開始前から存在した変更や、所有者を特定できない変更を含めない。
- `git add .`、`git add -A`、展開前に対象が確定しないワイルドカードを使わない。
  ステージするパスを確認し、`--` 以降へ個別に渡す。
- `--no-verify` を使わず、フックを回避しない。
- 常に新しいコミットを作成し、`git commit --amend`、`git reset`、`git rebase` を
  実行しない。作業ツリーとインデックスの状態を退避または破棄する `git stash`、
  `git restore`、`git checkout -- <path>`、`git clean` も実行しない。対象外の変更を
  コミットから分離する目的でも、これらの操作で退避せず、延期の規則に従う。
- `git push` を実行しない。`git fetch`、`git pull`、リモート設定の変更、リモート API に
  よるブランチ、タグ、PR、レビュー、マージ、リリースの作成や変更も行わない。この
  スキルではリモートと通信せず、ローカルのステージと新規コミットだけを変更する。
  同じ依頼にリモート操作が含まれていても、このスキルの権限として実行しない。
- リポジトリの指示や確認できる Git フックがリモート更新を行う場合は、フックを迂回せず
  コミットを延期する。フックが行う外部操作を確認できない場合も延期する。
- `write-commit-messages` が確定した完全な文面をそのまま使い、独自に整形または追記しない。
- 秘密値をチャット、ログ、コミットメッセージへ転載しない。疑わしいファイルは
  パスだけを報告する。

## 1. リポジトリと変更状態を確認する

リポジトリのルート、リポジトリ固有の指示、現在のブランチ、進行中の Git 操作を確認する。
可能なものは並行して実行し、少なくとも次を確認する。

- `git status --short`
- `git branch --show-current`
- `git diff --cached --name-status`
- `git diff --name-status`
- `git log --oneline -10`
- `git rev-parse --show-toplevel`、`git rev-parse --git-dir`、
  `git rev-parse --git-common-dir`
- `git config --show-origin --get core.hooksPath`

コミットがまだないリポジトリでは `git log` が失敗し、`core.hooksPath` が未設定の
リポジトリでは `git config --show-origin --get core.hooksPath` が非ゼロで終了する。
どちらも異常と扱わず、履歴がない、または既定のフックディレクトリを使うという
事実として記録し、他の確認を続ける。

リポジトリで最初に自動コミットする前とフック設定が変わった後は、有効なフックディレクトリの
実行可能な `pre-commit`、`prepare-commit-msg`、`commit-msg`、`post-commit` を確認する。
`core.hooksPath` が未設定なら、`git rev-parse --git-common-dir` で確認した共通 Git
ディレクトリ内の `hooks` を確認する。linked worktree では `git rev-parse --git-dir` が
worktree 固有のディレクトリを返し、Git はそこからフックを読み込まないため、
フォールバック先の判定に `--git-dir` を使わない。

リポジトリで最初に自動コミットする前は、`git config --get commit.gpgsign` などで
コミット時の署名が有効かも確認する。署名が有効で、パスフレーズ入力などの対話なしに
署名できることを確認できない場合は、非対話実行の `git commit` が停止し得るため、
コミットを延期して理由を記録する。

同じく最初の自動コミット前に、`git var GIT_COMMITTER_IDENT` と
`git var GIT_AUTHOR_IDENT` で、コミッターと作者の識別情報を解決できることを確認する。
解決できない場合は、`git config` の変更や環境変数の設定でエージェントが識別情報を
補わず、コミットを延期して理由を記録する。

未追跡ファイルを対象に含める場合は、`git ls-files --others --exclude-standard` で正確な
パスを確認する。パス一覧に機密情報の疑いがないことを確認してから、対象となり得る
ステージ済み差分と未ステージ差分を読む。

進行中の Git 操作は `git status --short` に現れないことがあるため、
`git rev-parse --git-dir` で確認した Git ディレクトリ直下の `MERGE_HEAD`、
`CHERRY_PICK_HEAD`、`REVERT_HEAD`、`REBASE_HEAD`、`rebase-merge`、`rebase-apply`、
`BISECT_LOG` の有無で判定する。merge、rebase、cherry-pick、revert、bisect などが
進行中の場合は、通常の新規コミットを作成せず、現在の作業へ制御を戻す。進行中の
Git 操作を、このスキルの判断で完了または中断しない。

`git branch --show-current` が空を返す detached HEAD 状態では、ユーザーがこの状態での
コミットを明示的に依頼した場合を除き、コミットを延期して通常の進捗報告または最終報告に
理由を残す。ブランチに属さないコミットは後続のチェックアウトで参照できなくなり得るため、
この状態の解消はユーザーの判断に委ねる。

暗黙呼び出しでは、現在の作業単位へ帰属できる差分だけを候補とする。ステージ済みである
という理由だけで対象にしない。明示的なコミット依頼で対象範囲が指定されていない場合だけ、
ステージ済み差分を既定の候補にする。

## 2. コミット対象を確定する

自動呼び出しでは、ユーザーの依頼、作業開始時に確認した状態、エージェントの編集記録、
現在の差分を照合して対象を確定する。パス、`type`、ステージ状態だけから変更の所有者や
目的を推測しない。

- 対象外のステージ済み変更が 1 件でもある場合は、`git commit` がそれも含めるため延期する。
  自動でステージを解除せず、現在のインデックスを維持する。
- 同じファイルにステージ済み変更と未ステージ変更がある場合は、両方が現在の作業単位へ
  帰属し、同じコミットへ含めることが明らかな場合だけ続行する。作業開始前からの部分ステージや
  帰属不明の差分を上書きする `git add` は行わない。
- 同じ未ステージファイル内に現在の作業と作業開始前からの変更が混在する場合は、ファイル全体を
  ステージせず延期する。ハンクの所有者を推測して自動分割しない。
- 未追跡ファイルは、現在の作業単位でエージェントが作成した場合、またはユーザーが対象として
  明示した場合だけ内容を確認する。
- 秘密鍵、認証情報、トークン、実値を含む `.env` など、機密情報の可能性があるファイルを
  検出した場合はコミットしない。`*.pem`、`*.key`、`*.p12`、`*.pfx`、`*.jks`、
  `*.keystore`、SSH 秘密鍵、`credentials.json`、`service-account*.json`、
  `.aws/credentials`、`.netrc`、`.git-credentials`、認証トークンを含む `.npmrc` や
  `.pypirc` などを警戒し、値を表示しない。
- OS やエディタの一時ファイル、ログ、キャッシュ、依存物、ビルド生成物は、リポジトリで
  追跡する根拠またはユーザーの明示指定がない限り追加しない。`.DS_Store`、スワップファイル、
  `*.log`、`__pycache__/`、`node_modules/`、一時ディレクトリなどを確認する。

ファイル名だけでは機密性や不要性を断定しない。`.env.example`、テスト用のダミー資格情報、
意図的に追跡する生成物などは、内容、リポジトリの方針、作業記録を根拠に判断する。

## 3. 原子的な単位へ整理する

変更の目的と依存関係に基づいてコミット単位を決める。メッセージの `type` や
ディレクトリが異なるという理由だけで分割しない。1 つの機能とそのテストや文書は、
同じ目的に必要なら同じコミットへ含める。独立して取り消せる目的は分ける。

インデックスが空で、複数の完了単位をファイル単位で安全に分けられる場合は、実行順を決め、
各単位を 1 件ずつステージしてコミットする。複数の独立した目的が既に同時にステージ済みの
場合は、自動でインデックスを書き換えずコミットを延期する。1 つのファイル内に複数の目的が
混在する場合も、ハンクを推測して分割しない。

## 4. メッセージと実行計画を確定する

各コミットについて、`write-commit-messages` へ次の根拠を渡してメッセージを作成する。

- ユーザーが説明した目的と対象範囲
- エージェントが行った作業と検証の記録
- リポジトリ固有の指示と関連する履歴
- そのコミットだけの差分
- ユーザー指定のメッセージがある場合は、その文面

ステージ済みのコミットでは `git diff --cached` を差分の正本とする。まだステージしていない
計画では対象パスの確認済み差分から仮案を作り、ステージ後に同じ内容であることを確認する。

コミットごとの目的、正確な対象パス、完全なメッセージ、実行順、対象外として残す変更を
内部の実行計画として確定する。ユーザーへ計画を提示して承認を求めない。メッセージの正確性、
破壊的変更、対象範囲を判断できない場合は、推測や質問で補わずコミットを延期する。

## 5. ステージしてコミットする

計画策定後に状態が変わっていないことを `git status --short` と対象差分で再確認する。
複数コミットでは、計画どおりに完了した直前のコミットによる変化だけを許容する。
それ以外の変化があれば、帰属と計画を再評価する。安全に再確定できなければ延期する。

既にステージ済みの単一コミットは、追加の `git add` を行わずに実行する。インデックスが
空で、未ステージまたは未追跡の変更を含める場合は、コミット 1 件分の正確なパスだけを
`git add -- <path>...` でステージする。

各コミットで次の手順を順に実行する。

1. `git diff --cached --name-status` が計画したパスだけを含むことを確認する。
2. `git diff --cached` を読み、計画した変更だけであることを確認する。
3. `git diff --cached --check` を実行する。失敗した場合、指摘箇所が Markdown の
   ハード改行など、ファイル形式またはリポジトリの規約で意図された空白であると
   確認できるときだけ続行する。確認できないときはコミットせず元の作業へ制御を戻す。
4. ステージ後の差分が仮案と異なる場合は、現在の作業単位に帰属する差分だけであることを
   確認し、`write-commit-messages` でメッセージを作り直す。帰属不明の差分があれば延期する。
5. `git write-tree` が返す `tree` オブジェクト ID を、確認済み内容の識別子として記録する。
6. 完全なメッセージをリポジトリ外の一時ファイルへ書き、メッセージをシェルコマンドへ
   展開せず、`git commit -F <message-file>` でコミットする。
7. 成否にかかわらず、その一時ファイルだけを削除する。
8. 成功後は `git rev-parse HEAD^{tree}` が記録した `tree` オブジェクト ID と一致すること、
   `git log -1 --format=%B` が確定したメッセージと一致すること、`git status --short` の
   残存変更が想定どおりであることを確認する。

手順 8 の確認が一致しない場合は、フックによる整形やメッセージの書き換えなど、確認できる
範囲で原因を特定する。作成済みのコミットを `git reset` や `git commit --amend` で修正せず、
後続のコミットを延期して、コミット ID と不一致の内容を作業記録と進捗報告に残し、
扱いを呼び出し元の作業へ委ねる。

フックまたはコミットが失敗した場合は `--no-verify`、`git reset`、`git commit --amend` で
回避せず、現在の状態と失敗内容を呼び出し元の作業へ返す。失敗原因の修正が元の依頼範囲に
含まれる場合は、作業側で修正と検証を行った後、このスキルを再評価する。複数コミットの
途中で失敗した場合も、成功済みコミットを戻さない。

## 6. 結果を記録して作業を続ける

コミットごとに短縮 ID、タイトル、含めた変更、検証結果を作業記録へ追加する。対象外として
残した変更と、延期したコミットがあれば理由も記録する。本来の依頼が続いている場合は、
コミット報告だけでワークフローを終了せず、次の作業へ進む。

ユーザーへの通常の進捗報告または最終報告では、作成したコミットと残存変更を簡潔に示す。
リモートへの反映を提案または実行しない。

## 完了条件

- 完了して検証可能な作業単位だけを、ユーザー確認なしでコミットしている。
- エージェントの作業へ帰属できる対象だけが含まれている。
- 各コミットが 1 つの論理的な目的にまとまっている。
- メッセージが `write-commit-messages` の指示と対象差分に一致している。
- フックを回避せず、コミット内容とメッセージを実行後に確認している。確認が一致しない
  場合も、履歴を書き換えずに記録している。
- 対象外の変更と既存のステージ状態を破壊していない。
- リモートと通信せず、リモートリポジトリを変更していない。
- 作成したコミットまたは延期理由を記録し、本来の作業を継続している。
