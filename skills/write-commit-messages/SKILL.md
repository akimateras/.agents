# コミットメッセージを作成する

## 目的

コミット対象の変更と目的を正確に表し、後から履歴を読む人と自動化ツールが
変更の要点を識別できるメッセージを作成、改訂、翻訳、レビューする。
形式は、第 1 節の規則で機械的に強制される規約を優先する場合を除き、
Conventional Commits 1.0.0 に従う。メッセージ全体を、第 3 節の優先順位で
選択した 1 つの言語で統一して書く。フッターや Git トレーラーは、
第 7 節と第 8 節の規則で認められる場合を除き使わない。

## 日本語で書く場合は必須規約を読む

選択した言語が日本語の場合は、メッセージを作成またはレビューする前に、
このファイルを基準に `../ja-tech-docs/references/style-ja.md` を全文読み、
その生成手順と完成後の日本語監査に従う。この日本語規約とこのスキルの規則が
矛盾する場合は、このスキルの規則を優先する。例えば `Changes:` 項目の末尾には、
日本語規約の箇条書きの規則にかかわらず句点を付けない。
日本語規約を利用できない場合は、独自の規則で代替せず、
メッセージを確定できないことを報告する。

## 依頼範囲を守る

- メッセージの提案、改訂、翻訳、レビューだけを求められた場合は、
  `git add`、`git commit`、`git commit --amend`、`git push` を実行しない。
- コミットの実行を明示的に依頼された場合だけ、メッセージを決めた後に
  コミットを別の操作として実行する。`commit-changes` スキルを利用できる場合は、
  ステージとコミットの操作をその手順に従って行う。対象外の変更をステージしない。
- コミットを実行するときに `commit-changes` を利用できない場合は、確定した
  メッセージをリポジトリ外の一時ファイルへ書いて `git commit -F` で渡し、
  シェルコマンドへの直接埋め込みで文面を崩さない。
- `git commit --amend`、`git rebase`、`git push` は、それぞれ明示的に
  依頼されない限り実行しない。
- メッセージに合わせるために実装やテストを変更しない。
- 差分から確認できない目的、Issue、互換性への影響、テスト結果を補完しない。
- マージコミットと、`fixup!` または `squash!` で始まるコミットは、
  Conventional Commits 形式を適用すると既定文面や自動スカッシュの仕組みと
  衝突するため、このスキルの対象外とする。これらのメッセージを依頼された
  場合は、対象外であることを報告する。

## 1. 対象と規約を確認する

次の順で、利用できる根拠を確認する。

1. ユーザーが指定した変更範囲、候補数、`type`、`scope`
2. リポジトリの指示、コントリビューションガイド、`commitlint` などの設定、
   コミットメッセージテンプレート
3. `git status --short` で確認できるステージ済み、未ステージ、未追跡の状態
4. `git diff --cached`、`git diff`、ユーザーが提示した差分または変更説明
5. `git log` と、必要に応じて `git show` で確認した同種のコミットの言語、
   `scope`、形式、情報量

過去のコミットは、第 3 節の言語選択の判断材料には使うが、明示的に人が確認した
表記規約がない限り、文体や語句の正本にしない。同じ表現が繰り返されていても、
頻度だけを理由に再利用しない。ただし過去のコミットで使われている `scope` は、
コードベースの構成要素の表記と矛盾しない場合に再利用できる。

ユーザーが指定した `type` と `scope` は、差分とこのスキルの規則に一致する場合だけ使う。
一致しない場合は黙って差し替えず、矛盾を報告したうえで適切な案を示す。
リポジトリの規約からは、`scope` の命名、タイトルの長さ、本文の書き方など、
このスキルと両立する規則を採用する。異なる形式を要求する規約が、
`commitlint` やコミットフックなどで機械的に強制され、従わないメッセージが
拒否されることを設定や文書で確認できる場合は、矛盾を報告したうえで、
その規約の要求を以降の節の形式規則より優先する。強制を確認できない規約との
矛盾は報告し、成果物ではこのスキルの形式を維持する。フッターと Git トレーラーの
追加可否は、第 7 節の規則だけで判定する。

実際にコミットするメッセージでは、ステージ済み差分を正本とする。
ステージ済み差分が空の場合は、未ステージ差分をコミット対象として黙って扱わない。
ユーザーまたは呼び出し元の手順がコミット対象を明示している場合は、その対象の
差分を正本とする。対象が明示されていない場合は、ステージ済みの変更がないことを
報告してコミット対象の指定をユーザーに確認する。
提案だけの依頼で未ステージ差分を使う場合は、何を根拠にした案か分かるようにする。
既存コミットを対象とする場合は、そのコミットの差分を正本とする。
複数のコミットを 1 つにまとめたコミットのメッセージでは、対象のコミット
全体を結合した差分を正本とする。明示的に依頼された `git commit --amend` で
ステージ済み変更を加える場合は、対象コミットの差分とステージ済み差分を
結合した差分を正本とする。
文面の翻訳または言い換えだけを求められ、差分がない場合は、与えられた意味を維持し、
差分との一致を確認したとは報告しない。

同じファイルにステージ済み変更と未ステージ変更がある場合は両方を確認し、
コミット対象外の内容をメッセージへ含めない。未追跡ファイルは、依頼範囲に関係する
場合だけ内容を確認し、メッセージ作成だけを理由にステージしない。

ロックファイルや生成物を含むなどで差分が非常に大きい場合は、`git diff --stat` などで
変更の範囲と規模を先に確認し、メッセージの根拠に必要な部分の内容を読む。
概要だけを確認した部分について、読んでいない詳細を推測して記載しない。

## 2. 変更の要点を整理する

- 変更されたファイルの一覧ではなく、コミットによって生じる結果を特定する。
- 何を変えたかに加え、差分やユーザー説明で確認できる場合は、なぜ変えたかを整理する。
- 出力文を書く前に、主要な変更を主体、対象、動作、条件、結果の形で整理する。
  差分、タイトルの下書き、過去のメッセージから語句のまとまりを転記しない。
- 実装方法より、利用者や保守担当者が履歴から識別したい動作、契約、運用上の変化を
  優先する。
- 無関係な目的が同じ差分に含まれる場合は、曖昧な総称でまとめず、コミット分割が
  必要であることを報告する。依頼に応じて各コミットのメッセージ案を示す。
- 破壊的変更、移行要件、非推奨化は、根拠がある場合だけ明示する。

## 3. 言語を選択する

メッセージ全体で 1 つの言語を使い、タイトル行の `description`、説明、重要な記録、
`Changes:` の各項目をすべて同じ言語で書く。ただし `type` は選択した言語にかかわらず
常に英語で書き、`scope` はコードベースの表記を選択した言語へ翻訳せずに維持し、
`Changes:` の見出しはこの表記のまま翻訳しない。

言語は次の優先順位で決める。

1. ユーザーが言語を指示した場合は、その言語を使う。
2. リポジトリの指示やコントリビューションガイドがメッセージの言語を明文で
   定めている場合は、その言語を使う。
3. どちらもなければ、`git log` で確認した直近のコミットの言語を使う。
   マージコミットと自動生成されたコミットを除外しながら履歴を遡り、
   除外されなかった直近 10 件を対象とする。遡って確認する履歴は直近 50 件
   までとする。10 件に達する前に履歴が尽きた場合や、50 件を確認しても
   10 件に達しない場合は、それまでに集めたコミットを対象とする。
   各コミットの言語は、`type`、`scope`、識別子を除いた `description` の言語で
   判定する。Conventional Commits 形式でないコミットは、識別子を除いた
   タイトル行全体の言語で判定する。対象の過半数を占める言語があれば、その言語を使う。
   自動生成されたコミットとは、作者名が `[bot]` で終わるなど作者を bot と
   確認できるコミット、`Merge` や `Revert` などの Git やホスティングサービスの
   既定文面のままのコミット、リリース自動化や依存関係更新ツールが作成した
   コミットを指す。
4. 過半数を占める言語がない場合や、参照できるコミットがない場合は、
   英語を使う。

どの言語を選択した場合も、次の規則を守る。

- 識別子、コマンド、API 名、製品名、規格名、Issue 参照は原表記を維持する。
- `scope` はコードベースの構成要素に使われている表記を優先する。
- 正式名称や識別子を、本文の言語へ合わせるためだけに翻訳しない。
- 英語で書く場合は、`description` と `Changes:` の各項目を命令形の動詞で始め、
  どちらも固有名詞などの原表記を除き小文字で始める。
- 日本語で書く場合は、正確な識別子をインラインコードで示し、一般概念は定着した
  日本語またはカタカナで表す。
- どの言語で書く場合も、別の言語で下書きした文の翻訳ではなく、
  整理した変更事実からその言語で直接作文する。翻訳の依頼では、
  原文から抽出した意味を変更事実として同じ手順で直接作文し、
  原文の語順や語句のまとまりを写さない。

## 4. Conventional Commits 形式を組み立てる

すべてのメッセージを次の構造にする。`scope`、`!`、本文は省略できる。
`scope` を省略する場合は括弧ごと省き、破壊的変更でなければ `!` を置かない。
フッターは、Conventional Commits で許可されていても、第 7 節の規則で
認められる場合と、第 8 節の規則でレビューの修正版に維持する場合だけ置く。

```text
<type>(<scope>)!: <description>

[optional body]

[optional footers]
```

- `type` は次節の一覧から 1 つ選び、小文字で記載する。
- `scope` には変更対象のパッケージ、モジュール、コンポーネントなどを表す
  安定した名詞を使う。有用な範囲を特定できない場合は省略する。
- `type` または閉じ括弧の直後に必要なら `!` を置き、その後にコロンと半角空白を置く。
- `description` は必須とし、選択した言語で変更対象と結果を簡潔に記載する。
- タイトルは 1 行に収める。リポジトリにタイトルの長さの規約がなければ、
  タイトル行全体を表示幅 72 桁以内とする。長さは `type`、`scope`、記号を含む
  行全体で数え、全角文字は 2 桁、半角文字は 1 桁として計算する。
- 本文は `description` の次に空行を 1 つ置いて始める。
- 破壊的変更では `!` を必ず付け、`description` で破壊的な変更内容を示す。
  互換性への影響や必要な移行を知らずに適用すると重大な支障が生じる場合は、
  根拠に基づいて重要な記録に記載する。
  破壊的変更は `!` と本文で表し、`BREAKING CHANGE:` フッターは、リポジトリが
  必須とすることを確認できる場合を除き使わない。

## 5. `type` を選ぶ

第 1 節の規則で強制される規約を優先する場合を除き、この一覧以外の `type` を
作らない。ファイルの種類だけでなく、変更の主目的で選ぶ。

| `type` | 使用条件 |
|---|---|
| `feat` | 利用者から見える機能を追加するか、意図した挙動または契約を変更する。バグ修正と性能だけの改善は除く。 |
| `perf` | 機能上の契約を変えず、実行時間、メモリ、I/O などの性能改善を主目的とする。 |
| `fix` | バグ、回帰、脆弱性など、意図または仕様に反する挙動を修正する。 |
| `style` | 空白、整形、セミコロンなど、コードの意味や生成結果に影響しない変更だけを行う。画面の見た目の変更には使わない。 |
| `refactor` | 外部から見える挙動と性能を意図的に変えず、`style` を超える内部構造や保守性の改善を行う。 |
| `test` | 製品コードの挙動を変えず、テストケース、フィクスチャ、テスト基盤、検証範囲を追加または変更する。 |
| `build` | ビルドシステム、パッケージング、コンパイラ設定、外部依存、マニフェスト、ロックファイルを変更する。 |
| `ci` | 製品やローカルビルドを変えず、CI/CD のワークフロー、設定、実行スクリプトを変更する。 |
| `docs` | README、ガイド、API 文書、コードコメントなど、文書だけを変更する。 |
| `revert` | 以前のコミットを取り消すコミットを作成する。`description` で取り消す変更を識別し、必要なら対象 SHA を説明に記載する。 |
| `chore` | 上記のどれにも分類できないリポジトリ保守を行う。最後の選択肢として使う。 |

境界が曖昧な場合は、次の基準で決める。

- 新しい挙動または意図した挙動変更は `feat`、期待される挙動の回復は `fix`、
  挙動を維持した内部変更は `refactor` とする。
- 性能改善が主目的なら `perf` とする。正しさの問題を直した結果として性能も改善する
  場合は `fix` とする。
- 既存の期待される挙動を回復するために製品コードを修正し、失敗していたテストを
  通す場合は `fix` とする。新しい挙動のテストを通すために製品コードを実装する場合は
  `feat` とし、誤っているテストコードや期待値だけを修正する場合は `test` とする。
- 空白や機械的な整形だけなら `style`、コード構造を変えるなら `refactor` とする。
- ローカルでも使うビルド処理や依存関係なら `build`、CI/CD だけの制御なら `ci` とする。
- 製品コードで脆弱性を修正する場合は `fix`、外部依存の更新だけで対応する場合は
  `build(deps)` とする。
- 機能変更に付随してテストや文書を更新する場合は、機能変更の `type` を使う。
  独立した目的が複数ある場合はコミットを分ける。
- 依存関係は `build(deps)`、リリース作業は `chore(release)` のように、
  新しい `type` ではなく既存の `type` と `scope` で表す。

## 6. 任意の本文の根拠を判定する

任意の本文は次の 3 ブロックをこの順序で使う。各ブロックは省略できる。
タイトルだけで変更の対象と結果を十分に伝えられる小さな変更では、
ブロックを埋めるための本文を作らず、本文全体を省略する。

| ブロック | 記載内容 | 使用できる根拠 |
|---|---|---|
| 説明 | 変更の目的、必要になった理由、発生していた問題などの背景情報と、整理した事実から選んだ重要な変更の概要および変更後の挙動 | 背景部分はユーザーの依頼や説明、Issue、仕様、観測した症状や診断結果、実作業の文脈。変更の説明は差分、実装、テスト、ユーザーの説明、実作業の結果。 |
| 重要な記録 | 重要な実装判断とトレードオフ、互換性への影響、旧バージョン環境へ適用する際の注意 | ユーザーの説明、仕様、作業中に確認した制約、実際に採用した判断とその結果。差分から直接確認できる互換性への影響に限り、差分だけを根拠にできる |
| `Changes:` | タイトルより細かい単位の変更点 | 差分、実装、テスト、実作業の結果 |

説明は 1 つの論理的な段落にする。背景情報に根拠がある場合は、背景情報を先に示し、
そのために行った変更の説明へ同じ段落内で続ける。背景情報に根拠がない場合は、
前置きを補わず、変更の説明から書き始める。

説明の背景部分と重要な記録には、差分以外の根拠を少なくとも 1 つ必要とする。
ただし、公開 API の削除やシグネチャの変更のように、差分から直接確認できる
互換性への影響は、差分だけを根拠として重要な記録に記載できる。移行手順や
判断の理由など、差分から読み取れない情報には引き続き差分以外の根拠を必要とする。
ユーザーが作成した変更についてメッセージだけを依頼し、エージェントが差分しか
確認できない場合は、説明には変更内容だけを記載し、差分から直接確認できる
互換性への影響を除いて重要な記録は省略する。
もっともらしい目的、問題、判断、トレードオフを差分から推測して補わない。
互換性への影響も、差分から直接確認できる範囲を超えて推測しない。

判定は変更の作成者ではなく、利用できる根拠に基づける。作成者にかかわらず、
差分しか確認できなければ、差分から直接確認できる互換性への影響を除き、
説明の背景部分と重要な記録を省略する。ユーザーが作成した変更でも、目的や判断を
ユーザーが説明していれば、その説明を根拠として使用できる。

同じエージェントが実作業を行った場合は、ユーザーの依頼、実際に観測した問題、
診断結果、明示的な制約、採用した実装判断、検証結果を根拠として背景情報と
重要な記録を検討できる。情報を知っているだけで記載を必須としない。内的な推論過程や、
作業後に作った理由を根拠にしない。会話や作業記録に根拠が残っていない場合は、
実作業を行ったと仮定せず省略する。

重要な記録は、根拠があることに加え、次のいずれかに該当して省略できない場合だけ書く。

- 知らずに既存環境へ適用すると、停止、データ損失、互換性問題、移行失敗などの
  重大な支障が生じる。
- 将来も維持すべき当時の判断、制約、またはトレードオフをコード、差分、テストだけから
  読み取れず、記録がないと保守時に誤った変更を招く。

変更内容の補足、通常の実装方法、形式の選択、規格への適合、検証結果は、重要な記録を
作る理由にしない。変更の理解に必要なら説明に含め、不要なら省略する。

任意ブロックを埋めるためだけにユーザーへ確認しない。正しい `type` の選択、
破壊的変更の判定、またはメッセージの正確性に必要な場合だけ追加情報を求める。

## 7. タイトルと任意の本文を書く

変更内容は次の順で作成する。この作成順は、メッセージでの出力順とは異なる。

1. 第 2 節の規則に従って整理した変更事実を確認する。
2. 整理した事実から重要なものを選び、選択した言語で説明を独立して作る。
3. 同じ事実から、タイトルより細かい変更点を `Changes:` として作る。
4. 主結果を 1 つ選び、タイトルへ圧縮する。

変更内容の詳細度と情報量は、`Changes:`、説明、タイトルの順に小さくする。
説明を `Changes:` の各項目の言い換えの列挙にしたり、全項目を 1 文へ連結したりしない。
重要な記録はこの作成順と圧縮階層に含めない。前節の条件を満たす場合だけ、
その固有情報を独立した段落として書く。
圧縮するときは重要度の低い事実を省き、文法要素を削って不自然な文を作らない。

タイトルと任意の本文は次の順で出力する。

```text
<type>(<scope>)!: <description>

<背景情報があれば先に置き、重要な変更の説明へ続ける段落。
必要に応じて単一改行で折り返す>

<絶対に必要な場合だけ記載する重要な記録の段落。
必要に応じて単一改行で折り返す>

Changes:
- <change>
- <change>
```

- 3 つの本文ブロックはそれぞれ省略できる。省略したブロックの見出し、説明、
  プレースホルダー、余分な空行を残さない。
- 最初の 2 ブロックには見出しを付けず、それぞれ 1 つの論理的な段落として書く。
  箇条書き、番号付きリスト、表を使わない。
- 1 つの論理的な段落を複数の文に分けてよい。条件、動作、結果の関係が曖昧になる
  場合は、1 文へ圧縮しない。
- 1 つの段落内では、選択した言語として自然な句読点を使う。可読性のために単一改行を
  入れてよく、単一改行で区切られた行は同じ段落として扱う。
- 段落を 1 行に収めることを優先しない。1 行がおおむね表示幅 72 桁を超えないよう、
  文または意味の切れ目で単一改行を入れる。
- 段落と段落、タイトルと本文、本文と `Changes:` は、空行 1 つ、つまり連続する
  2 つの改行文字で区切る。段落内には空行を入れない。
- 説明では、背景情報と変更内容を別の段落に分けない。背景情報がなければ、段落を
  埋めるための目的や問題を作らず、変更内容だけを書く。
- 説明では、すべての `Changes:` 項目を言い換えない。変更全体を理解するために
  重要な項目だけを選び、複数の細かい変更を 1 つの結果または方針として要約する。
- `Changes:` は、タイトルだけでは表しきれない独立した変更点が複数ある場合だけ
  追加し、直下に `- ` で始まる 1 階層のリストを重要度の高い順に置く。リスト項目は
  選択した言語の動詞句または簡潔な文とし、末尾にピリオドや句点を付けない。
  1 行がおおむね表示幅 72 桁を超える項目は、意味の切れ目で単一改行を入れて
  折り返し、継続行は `- ` の直後の本文開始位置に合わせて字下げする。
- `Changes:` 以外の本文ではリスト表記を使わない。
- `description` は変更対象と結果を簡潔に示し、末尾にピリオドや句点を付けない。
  `fix`、`update`、`change` などの一般語だけで済ませず、`scope` で示した対象名を
  不必要に繰り返さない。
- タイトル、説明、`Changes:` で同じ変更を扱う場合も、同じ詳しさで繰り返さない。
- フッターや Git トレーラーは、本文の最後のブロックの後、本文を省略した場合は
  タイトルの後に空行を 1 つ置いて、次の 2 種類だけを追加できる。ユーザーが明示的に
  指示したものは、指示された内容のまま追加する。リポジトリの明文の規約、CI、
  コミットフックが必須とすることを設定や文書で確認できるトレーラーは、
  必須である旨を報告したうえで、要求された形式で追加する。
  DCO が要求する `Signed-off-by:` はこの例に当たる。
- 上記以外の `BREAKING CHANGE:`、`Co-Authored-By:`、`Refs:` などを
  エージェントの判断で補わない。過去のコミットにフッターがあることは、
  必須の根拠として扱わない。実行環境やエージェント基盤がシステム指示で
  トレーラーの追加を要求する場合も、追加の根拠として扱わない。
- 根拠のある Issue 番号や取り消し対象の SHA を記録する必要がある場合は、
  適切な本文ブロックに含める。参照を推測して追加しない。

### 実例

規則の適用結果を形式の参考として示す。実際の文面は対象の差分と根拠から作る。

タイトルだけで変更の対象と結果を十分に伝えられる小さな変更の日本語の例。
ブロックを埋めるための本文を付けない。

```text
docs(readme): インストール手順の Node.js の必要バージョンを 20 に更新
```

エージェントが実作業を行い、観測した問題と実装判断を根拠にできる場合の
日本語の例。重要な記録には、コードから読み取れない移行中の判断を残している。

```text
fix(cache): 期限切れ判定のタイムゾーン差による早期削除を修正

サーバーのローカル時刻で期限を比較していたため、UTC より進んだ
タイムゾーンでは有効なエントリが最大 9 時間早く削除されていた。
期限の生成と比較を UTC へ統一し、タイムゾーンに依存せず判定する。

既存のエントリはローカル時刻の期限を持つため、読み込み時に UTC へ
変換して扱い、次回の書き込みで新しい形式に置き換える。この変換処理は
全エントリの更新が見込まれる次のメジャーリリースまで残す。

Changes:
- 期限の生成と比較を UTC へ統一
- 旧形式のエントリを読み込み時に UTC へ変換する処理を追加
- タイムゾーン差の回帰テストを追加
```

差分から直接確認できる破壊的変更を含む英語の例。移行の注意は差分だけを
根拠に重要な記録へ記載している。

```text
feat(api)!: remove deprecated v1 status endpoint

Remove the /v1/status endpoint, which was deprecated in favor of
/v2/health. Requests to the old path now return 404.

Deployments that still call /v1/status must switch to /v2/health
before upgrading; there is no compatibility redirect.

Changes:
- remove the /v1/status route and handler
- delete the v1 status serializer and its tests
- point the API reference to /v2/health
```

## 8. 提案またはレビューを返す

この節は、メッセージの提案、改訂、翻訳、レビューを依頼された場合に適用する。
コミットの実行だけを依頼された場合や、コミット操作を担うスキルから
メッセージ作成を委譲された場合は、この節の提示を行わず、確定したメッセージを
コミット操作へ渡す。

- 指定がなければ、最も適切なメッセージを 1 案だけコードブロックで示す。
- 複数案を求められた場合は、単なる言い換えではなく、正確さを保ったまま
  焦点または詳細度が異なる 2〜3 案を示す。
- 前提や対象差分に重大な曖昧さがある場合と、第 1 節に従い未ステージ差分などの
  根拠を明示する場合だけ、案の前後に短い注記を付ける。
- 具体的な案を作る情報が不足している場合は、対象差分または変更の目的を確認する。
- 既存メッセージのレビューでは、差分との一致、`type`、`scope`、言語の選択と統一、
  本文の順序、各情報の根拠を確認し、問題があれば修正版を示す。
- 対象の差分を参照できないレビューでは、形式、言語、構成など文面だけで確認できる
  規則をレビューし、差分との一致を確認していないことを注記する。
- レビューでは、既存のフッターや Git トレーラーを欠陥として扱わず、修正版でも
  原文のまま維持する。第 7 節の追加の条件は、トレーラーを新しく追加する場合にだけ
  適用する。

## 完了条件

- すべてのメッセージが、第 1 節の規則で優先した強制規約がある場合はその形式に、
  それ以外は Conventional Commits 形式になっている。
- `type` が定義済みの一覧、または第 1 節の規則で優先した強制規約の一覧にあり、
  変更の主目的と一致している。
- 差分が利用できる場合は、メッセージが対象の差分だけを表している。
- メッセージ全体が、優先順位に従って選択した 1 つの言語で統一されている。
- `type` が選択した言語にかかわらず英語で、`scope` がコードベースの表記のままで、
  `Changes:` の見出しが翻訳されずにこの表記のままである。
- タイトルが 1 行に収まり、その長さが規約または既定の上限を超えていない。
- 日本語で書いた場合は、メッセージがこのスキルの規則と矛盾しない範囲で共通の
  日本語規約に従い、残した英字表記を 1 件ずつ説明できる。
- 英語で書いた場合は、`description` と `Changes:` の各項目が命令形の動詞で始まり、
  固有名詞などの原表記を除いて小文字で始まっている。
- 本文を整理した変更事実から選択した言語で直接作成し、別の言語の下書きを
  直訳していない。翻訳の依頼では、原文の意味を維持したまま対象言語で
  自然に作文している。
- `scope` がコードベースの表記と、このスキルと両立するリポジトリの慣例に従っている。
- `description` から変更対象と結果を識別できる。
- 破壊的変更に `!` があり、`description` が破壊的な変更内容を示している。
- フッターと Git トレーラーが、ユーザーが明示的に指示したもの、リポジトリが
  必須とすることを確認したもの、レビューで維持した既存のものだけである。
- 任意の本文が定義済みの順序になり、`Changes:` 以外でリストを使っていない。
- タイトルだけで十分に伝えられる小さな変更に、本文を付けていない。
- 変更内容の情報が `Changes:`、説明、タイトルの順に圧縮され、説明が `Changes:` の
  全項目を言い換えまたは列挙していない。
- 背景情報と変更の説明が 1 つの段落にまとまり、背景情報がなければ説明だけになっている。
- 段落内の単一改行と段落間の空行が区別され、選択した言語の句読点が自然に
  使われている。
- 重要な記録に省略できない理由があり、差分から直接確認できる互換性への影響を
  除いて差分以外の根拠がある。条件を満たさない記載は省略されている。
- 各本文ブロックが、タイトルや他のブロックを冗長に繰り返していない。
- 根拠のない目的、Issue、テスト結果、互換性への影響を記載していない。
- メッセージに合わせるための実装やテストの変更を行っていない。
- 明示的な依頼なしに、ステージ、コミット、履歴の書き換え、リモートへの反映を
  行っていない。
